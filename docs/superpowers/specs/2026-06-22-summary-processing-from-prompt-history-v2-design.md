# Summary Processing from Prompt History V2 Design

## Goal

Use the existing 10-turn multi-turn Q&A result in
`tmp/prompt_history_eval_results_v2.json` as direct input to the current
structured summary generation path, then inspect whether the existing summary
chain can still produce a useful and faithful structured summary on more
naturalistic dialogue.

This step is intentionally narrower than full offline evaluation. The immediate
question is not whether the sample passes a formal suite. The immediate question
is whether the current summary path can ingest this dialogue and preserve the
right state.

## Scope

This step includes:

- reading the saved Q&A result file
- converting the 10 turns into summary `source_messages`
- running the existing structured summary generation path once
- saving and reviewing:
  - structured summary
  - rendered summary
  - raw model output
  - validation result

This step does not include:

- authoring a new formal `summary` eval sample
- running `eval-runner`
- adding or changing summary scoring rules
- changing production prompt logic before observing the first result

## Input

Primary input file:

- `tmp/prompt_history_eval_results_v2.json`

Input mapping:

- each `question` becomes one `user` message
- each `answer` becomes one `assistant` message
- messages remain in original turn order
- `previous_summary` is omitted for the first run

## Execution Path

Use the existing summary core path rather than the eval runner.

Preferred entry:

- `internal/app/rag/core/history.GenerateStructuredSummary(...)`

Relevant wrapper:

- `internal/app/rag/evaluation.NewHistorySummaryGenerator(...)`

Rationale:

- this is the narrowest path that exercises the real structured summary
  generation logic
- it avoids premature eval-sample authoring work
- it returns exactly the artifacts needed for a first-pass manual review

## Model Strategy

### Primary model

Use `qwen-max` first.

Reason:

- the source Q&A was already generated by `qwen-max`
- this isolates whether the current summary chain can handle this dialogue shape
  before introducing cross-model style differences

### Fallback / comparison model

Use `glm-5` only if the first result is ambiguous or clearly unstable.

Reason:

- it is useful as a style/control comparison
- it should help distinguish model-style issues from summary-chain issues

### Not preferred for first pass

Do not start with `deepseek-r1`.

Reason:

- it is more likely to over-explain or over-expand summaries
- that makes first-pass diagnosis noisier

## Output Artifacts

The run should preserve four artifacts:

- structured summary JSON
- rendered summary text
- raw model response
- validation result

These should be saved to a local temp artifact file so the output can be
reviewed without rerunning the model call.

## Review Checklist

First-pass review should focus on these questions:

1. Does `goal` capture the real thread?
   - building an enterprise knowledge-base QA system
   - making architecture decisions around retrieval and answer quality

2. Does `constraints` preserve the key real constraint?
   - PostgreSQL is already the main store
   - the team does not want to operate a separate retrieval system too early

3. Does `established_facts` avoid turning discussion into committed decisions?
   Examples to watch:
   - proposed use of `pgvector`
   - adding BM25 / hybrid retrieval
   - using RRF
   - where rewrite should sit

4. Does the summary over-compress the later turns?
   Important late-turn content includes:
   - rewrite risk
   - retrieval evaluation metrics
   - scale-up concerns
   - incident-style fault isolation across rewrite / recall / rerank / context /
     generation

5. Are open questions still represented as open?
   The dialogue is exploratory. The summary should not flatten uncertainty into
   false certainty.

## Recommended Minimal Run Order

1. Read `tmp/prompt_history_eval_results_v2.json`
2. Build ordered `source_messages`
3. Run structured summary generation with `qwen-max`
4. Save the four output artifacts
5. Review manually against the checklist above
6. Only if needed, rerun with `glm-5`

## Success Criteria for This Step

This step is successful if:

- the summary path runs without code changes to production summary logic
- the structured summary remains coherent on this more naturalistic dialogue
- we can clearly tell whether the current path is:
  - broadly usable
  - lossy but fixable
  - fundamentally drifting on this input style

## Risks

### Risk 1: discussion becomes fake certainty

The model may summarize proposed architecture as already-final implementation
fact.

### Risk 2: late-turn compression loss

The model may preserve early architecture content but drop later diagnostic and
evaluation details.

### Risk 3: overly generic goal

The model may collapse the thread into a generic goal like “design a QA system”
and lose the retrieval/quality focus.

### Risk 4: misleadingly good rendered text

The rendered summary may read smoothly while the structured fields are actually
missing important distinctions. Structured output should remain the primary
artifact under review.
