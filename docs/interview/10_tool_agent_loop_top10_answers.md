## 1. 为什么需要 `Behavior`，不用它会怎样？

**参考回答**

`Behavior` 的核心价值，是把“工具执行结果”翻译成“Agent 运行时可以消费的语义”。  
如果没有它，系统拿到的只是一个原始 `Result`，知道工具成功执行了，也知道返回了哪些字段，但并不知道这些字段在运行时意味着什么。

具体来说，没有 `Behavior`，系统会缺 4 类能力：

1. 无法稳定判断当前证据够不够回答
2. 无法从结果里自然地续推出下一步调用
3. 无法把 raw result 转成更适合 prompt 的上下文
4. 无法在最终回答里显式表达证据优先级、置信度和不确定性

如果强行不加这一层，通常只会走向两个坏结果：

1. 把大量业务语义堆到 `AgentLoop` 或 `Observer` 里，最后变成中心化大 `switch`
2. 把所有语义判断都交给 LLM，每一轮重新理解结果，成本高且不稳定

所以 `Behavior` 其实是在做一层职责分离：

- `Invoker` 负责“怎么查”
- `Behavior` 负责“查完意味着什么”

这层一旦拆出来，tool 才不只是“能被调用”，而是“能被 Agent 稳定消费”。

**回答结构**

1. 先定义 `Behavior` 是“语义层”
2. 再讲没有它会缺什么
3. 最后讲两种坏后果：中心化逻辑或全交给 LLM

## 2. 一个具体 tool 从注册到最终回答，完整调用链是什么？

**参考回答**

我一般会用 `document_ingestion_diagnose` 或 `document_query` 举例。

完整链路可以按 6 段来讲：

1. 定义  
   先定义 tool 的输入参数、执行逻辑和输出结果结构，比如输入 `documentId`，返回状态、结论、节点信息等。

2. 组装  
   给这个 tool 配上 `Spec` 和 `Behavior`，然后在 assembly 层注册成 `ToolModule`。

3. 选择  
   在某一轮里，要么由 planner 选中它，要么由上一轮 observer 产出的 `NextHintCalls` 指向它。

4. 执行  
   `Executor` 根据 registry 取到这个 module，调用 invoker，得到标准化 `Result`。

5. 消费  
   `Observer` 结合 `Behavior` 判断：
   - 当前结果是否足够
   - 是否继续下一轮
   - 下一步调用哪个 tool
   - 当前 `AgentState` 怎么更新

6. 回答  
   `AgentLoop` 收尾时把所有结果交给 `RenderContext` 和 `BuildGuidance`，生成 `ToolContext` 和 `AnswerGuidance`，再进入 prompt，最后由模型组织答案。

所以一个 tool 的生命周期不是“执行一次函数”，而是：

定义 -> 注册 -> 被选中 -> 被执行 -> 被观察 -> 被渲染进最终回答

**回答结构**

1. 选一个具体 tool
2. 按时间顺序讲 6 段
3. 最后强调“最终回答消费”这一段

## 3. 你们的 `Plan-Act-Observe` 是怎么运转的？

**参考回答**

这套机制本质上是一个带状态的多轮闭环，而不是单轮 function calling。

每一轮的职责可以拆成三段：

1. `Plan`  
   根据用户问题、已有工具结果和当前 `AgentState`，决定本轮调用哪些 tool。

2. `Act`  
   执行这些工具，得到标准化 `Result`。

3. `Observe`  
   消费这一轮结果，判断：
   - 当前证据是否已经足够
   - 当前假设是什么
   - 下一轮是否继续
   - 如果继续，下一步 hint 是什么

观察阶段的输出会写回 `AgentState`，再进入下一轮 planning。  
所以它不是“多调几次工具”，而是“根据上一轮证据逐轮下钻”。

我通常会把它理解成一个证据驱动的 runtime：

- `Plan` 决定往哪里查
- `Act` 负责把事实查出来
- `Observe` 决定这些事实下一步该怎么用

**回答结构**

1. 先和单轮调用做区分
2. 再分三段解释
3. 最后强调 `AgentState` 的承接作用

## 4. `AgentState` 和单轮 `Result` 的关系是什么？

**参考回答**

`Result` 和 `AgentState` 的职责不一样。

- `Result` 是单次工具调用返回的局部事实
- `AgentState` 是多轮运行时的全局状态快照

单个 `Result` 只能说明“这一次查到了什么”，但它不能直接表达：

- 当前链路处于什么阶段
- 当前最可信的 hypothesis 是什么
- 已经检查过哪些工具
- 还有哪些未解问题
- 下一轮应该优先调用什么

这些跨轮次信息必须放在 `AgentState` 里。

所以我会把它们的关系理解成：

- `Result` 是原材料
- `AgentState` 是 runtime 对这些原材料的阶段性归纳

没有 `Result`，就没有事实来源；没有 `AgentState`，多轮链路就会每一轮都像白纸一样重新判断。

**回答结构**

1. 先区分“局部事实”和“全局状态”
2. 再列出只有 state 才能表达的内容
3. 最后用一句比喻收口

## 5. 为什么不是把所有下一步决策都交给 LLM？

**参考回答**

因为“完全交给 LLM”在复杂工具链里通常不是最优解。

它的问题主要有三类：

1. 成本高  
   多轮诊断链里，如果每一轮都让 LLM 重新理解所有结果并决定下一步，调用成本会很快上升。

2. 稳定性差  
   结构化问题其实有很多稳定的续推模式，比如从 document 到 task，再到 node，下游路径并不需要每轮都重新推理。

3. 容易幻觉  
   尤其是 observer 这类角色，如果没有足够约束，可能产出并不存在的 tool 或错误的实体 ID。

所以我们做的是混合决策：

- 高层规划、首轮理解，交给 LLM
- 稳定续推、结构化下钻，交给规则和 behavior
- LLM 产出异常时，再回退到规则路径

这不是“减少智能性”，而是在做职责分工，让 LLM 用在更值钱的地方。

**回答结构**

1. 先讲全 LLM 的 3 个问题
2. 再讲混合策略
3. 最后收口到“职责分工”

## 6. 规则路径和 LLM 路径各自适合什么场景？怎么做 fallback？

**参考回答**

这两条路径各有擅长场景。

规则路径更适合：

- 结构化 ID 明确的问题
- 诊断链已经进入稳定下钻阶段的问题
- 已有结果可以直接续推的问题

LLM 路径更适合：

- 用户表达很开放或很模糊的问题
- 首轮需要理解意图和路由的问题
- 单靠固定规则很难覆盖的问题

我们的 fallback 思路通常是：

1. 优先尝试更高价值的 LLM 理解能力
2. 如果 LLM 失败、解析失败、hint 不合法或缺乏证据支撑
3. 就回退到规则观察或 behavior 驱动续推

所以 fallback 的目标不是“兜底一个错误”，而是“把不稳定路径重新拉回可控路径”。

**回答结构**

1. 先分别讲两条路径适合什么
2. 再讲 fallback 触发条件
3. 最后讲 fallback 的真正目的

## 7. `RenderContext` 和 `BuildGuidance` 的区别是什么？

**参考回答**

这两个都是结果消费阶段的能力，但解决的问题不同。

`RenderContext` 更偏“事实组织”：

- 它负责把 tool 的执行结果整理成适合 prompt 消费的上下文
- 重点是把 raw result 转成清晰、可引用、可归纳的事实块

`BuildGuidance` 更偏“回答约束”：

- 它负责告诉最终模型应该怎么用这些事实来回答
- 比如优先说结论、如何表达不确定性、冲突证据以谁为准

所以我会把它们区分成：

- `RenderContext`：给模型看什么
- `BuildGuidance`：要求模型怎么说

这两条通道拆开之后，运行时就不会把“事实本身”和“回答策略”混在一起。

**回答结构**

1. 先说两者都属于结果消费
2. 再对比“事实组织”和“回答约束”
3. 最后用一句短对比收尾

## 8. 如果一个 tool 成功执行了，但没有触发后续诊断，你怎么排查？

**参考回答**

这类问题我会优先当成“silent degradation”来查，也就是调用没报错，但语义链路断了。

排查时我会按 4 层顺序看：

1. 数据契约  
   看 `Result.Data` 有没有缺关键字段，比如 `documentId`、`status`、`latestTaskId` 这类后续续推依赖字段。

2. 行为层  
   看 `Behavior.Next` 或 `Behavior.Observe` 有没有正确产出 `NextHintCalls` 和新的状态。

3. 运行时层  
   看 `AgentLoop` 有没有正确消费新的 state，planning 是否真的使用了这些 hint。

4. 可观测性层  
   看日志和 trace 里，当前轮是 planner 没选中、observer 没续推，还是 hint 被过滤掉了。

这类问题很多不是“工具没跑”，而是“结果没有被 runtime 正确消费”。

**回答结构**

1. 先定义问题类型是 silent degradation
2. 再按 4 层排查
3. 最后点出关键判断：不是调用失败，是消费失败

## 9. 多轮 `AgentLoop` 的主要性能开销在哪？你们做过哪些优化？

**参考回答**

多轮 `AgentLoop` 的主要性能开销通常不在本地状态维护，而在两类地方：

1. LLM 调用  
   包括 planner 和 observer 的推理成本

2. 外部 I/O  
   包括工具查库、查 trace、查网页、抓取外部内容等

我们做过的优化里，比较关键的是把多轮里的 planner 冗余调用压掉。  
具体做法是：

- 首轮仍允许 LLM 参与规划
- Round 2 以后如果已经有结构化 `NextHintCalls`
- 就直接按 hint 续推，跳过重复 planner

这个优化的收益很明确：

- 降低多轮链路的 LLM 成本
- 缩短稳定诊断路径的响应时间
- 保持结果链路不回退

如果继续优化，我会继续从这两个方向下手：

- 让更多稳定路径少走 LLM
- 让高延迟 I/O 工具有更清晰的并行和短路策略

**回答结构**

1. 先讲两大开销来源
2. 再讲一个具体优化
3. 最后讲未来优化方向

## 10. 你们怎么证明这不是“能跑”，而是真的“可维护、可扩展、可观测”？

**参考回答**

我会从三个维度来证明。

第一，可维护。  
我们把工具拆成 `Invoker / Spec / Behavior`，避免把执行逻辑、业务语义和运行时控制写在一起，这样后续改某个 tool 时，不需要频繁改中央循环。

第二，可扩展。  
新工具接入时，按 module-first 模式接入 registry 和 assembly，不需要在 `AgentLoop` 里加一大堆特例。也就是说，扩展点主要在 module，而不是中央 runtime。

第三，可观测。  
每轮都会记录 planning source、tool call、observer 决策、state 演化和 degrade reason，能区分到底是 planner、tool、observer 还是 prompt 构造出了问题。

如果一套系统只是“能跑”，它通常一遇到新工具、新链路或者线上质量问题就会迅速失控。  
而我们现在这套框架的价值，是它已经有比较清晰的职责边界、扩展方式和排障抓手，这才叫真正能支撑持续演进。

**回答结构**

1. 先按“可维护、可扩展、可观测”三维展开
2. 每个维度讲一个机制
3. 最后用“持续演进能力”收口

## 最后建议

这 10 个问题里，至少要练熟下面 5 个：

1. 为什么需要 `Behavior`
2. 一个 tool 的完整调用链
3. `Plan-Act-Observe` 如何运转
4. 为什么不是全 LLM 决策
5. 如何证明这套框架不只是“能跑”

如果这 5 个问题你都能稳定答到 1 到 2 分钟，并且能接住追问，这个模块的技术面基本就不会虚。
