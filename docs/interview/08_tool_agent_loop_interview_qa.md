## 开场 30 秒版本

如果面试官让你先概述这段经历，可以先用这个版本：

> 我做的这块不是单纯加几个 function calling，而是搭了一套统一的 tool runtime。它把查询、诊断、网页检索等工具统一纳入注册、执行和结果消费链路，再在外层用 Plan-Act-Observe 的多轮机制，让系统能根据上一轮证据继续下钻定位问题，或者在知识不足时自动补充外部证据。核心价值是把工具从“能调用”提升到“能被 Agent 稳定编排和消费”。

**回答思路**

- 先讲不是“几个工具函数”
- 再讲“统一 runtime + 多轮机制”
- 最后点明业务价值

## 核心问答

### Q1. 为什么要搭一套统一的 tool runtime？

**参考回答**

因为业务一旦进入多工具、多轮诊断和外部证据补充场景，问题就不再是“能不能调工具”，而是“工具能不能被统一编排和统一消费”。  
如果工具是散落的函数调用，会很快遇到几个问题：

- 注册方式不一致
- 参数和结果结构不一致
- 后续决策逻辑分散
- 很难接入统一的可观测性和 prompt 构造

所以我做这套 runtime 的目标，不是把调用包一层壳，而是统一工具从定义、注册、执行到结果消费的完整生命周期。

**回答思路**

1. 从业务复杂度上升讲起
2. 说明“调用问题”和“消费问题”不是一回事
3. 最后强调“统一生命周期”

### Q2. 这套框架的核心抽象是什么？

**参考回答**

我们最后的主抽象是 `ToolModule`。  
它把三层东西组合到一起：

- `Invoker`：负责真正执行工具
- `Spec`：负责能力域、风险、证据来源、执行模式等元数据
- `Behavior`：负责结果怎么被 Agent 理解和消费

在它外面还有三个运行时角色：

- `Registry`：统一注册和索引这些 module
- `Executor`：统一执行工具
- `AgentLoop`：统一做多轮 Plan-Act-Observe 编排

所以这个模块的重点不是某个单一 tool，而是一套 module-first 的运行时模型。

**回答思路**

1. 先给出主抽象
2. 再拆三层职责
3. 最后补运行时外围角色

### Q3. 为什么要把 `Invoker` 和 `Behavior` 分开？

**参考回答**

因为这两个问题本质上不是一类问题。

- `Invoker` 解决的是“怎么查”
- `Behavior` 解决的是“查完意味着什么”

比如一个工具查到了 `document status=failed`，这本身只是原始结果。  
系统还需要知道：

- 这个结果够不够回答
- 是否还要继续下钻
- 下一步该调用哪个 tool
- 最终 prompt 里怎么组织这份结果

这些都不属于执行逻辑，所以必须从 invoker 里拆出来，放进 behavior。

**回答思路**

1. 先讲职责边界
2. 再用一个结果示例说明“查完不等于会用”
3. 用一句对比收口

### Q4. `Behavior` 具体解决了什么问题？

**参考回答**

`Behavior` 主要解决四类问题：

1. 结果续推  
   比如根据当前结果判断下一步 hint 是什么

2. 轮次观察  
   比如判断这一轮该停还是继续，并产出新的 `AgentState`

3. 上下文渲染  
   比如把 raw result 变成更适合 prompt 消费的 `ToolContext`

4. 回答指导  
   比如告诉最终模型应该优先强调哪些证据、如何表达不确定性

所以我会把它定义成“tool 和 agent runtime 之间的语义适配层”。

**回答思路**

1. 不要只说“behavior 很重要”
2. 尽量拆成 4 个具体能力
3. 最后给一个高层定义

### Q5. 一个 tool 从定义到最终回答，完整链路是什么？

**参考回答**

我用 `document_ingestion_diagnose` 举例会比较清楚。

完整链路是：

1. 定义 tool 的参数和执行逻辑
2. 给它配置对应的 `Behavior`
3. 在 assembly 层注册成 `ToolModule`
4. `Registry` 保存这个 module
5. `Planner` 或者上一轮 hint 选中这个 tool
6. `Executor` 执行 tool，拿到 `Result`
7. `Observer` 结合 behavior 生成新的 `AgentState`
8. 如果证据不足，继续下一轮；如果足够，就停止
9. 收尾阶段把所有结果交给 `RenderContext`
10. 再生成 `AnswerGuidance`
11. 两者一起进入 prompt，组织最终回答

所以对我来说，一个 tool 的生命周期不是“被调了一次”，而是“被定义、被执行、被解释、被回答消费”。

**回答思路**

1. 选一个具体 tool
2. 一定按时间顺序讲
3. 最后强调“最终回答消费”

### Q6. 你们的 Plan-Act-Observe 怎么运作？

**参考回答**

这是一个带状态的多轮闭环。

- `Plan`：根据问题、已有结果和当前状态决定本轮调用什么
- `Act`：执行工具，得到结构化结果
- `Observe`：判断当前证据是否足够、下一轮是否继续、下一步 hint 是什么

观察阶段的输出会沉淀到 `AgentState` 里，再进入下一轮 planning。  
所以这套机制的价值，不是“多调几次工具”，而是“根据证据逐轮推进问题定位”。

**回答思路**

1. 先概括成闭环
2. 再讲三段职责
3. 最后强调 `AgentState` 的承接作用

### Q7. 为什么需要 `AgentState`？

**参考回答**

因为多轮系统最怕每一轮都像白纸一样重新推理。  
`AgentState` 的作用，就是把上一轮观察出来的关键信息保留下来，例如：

- 当前阶段 `phase`
- 当前假设 `hypothesis`
- 置信度 `confidence`
- 已检查工具
- 未解问题
- `NextHintCalls`

这样下一轮 planning 不需要完全重算，而是可以沿着上一轮证据继续下钻。

**回答思路**

1. 先讲没有 state 会怎样
2. 再讲 state 里保存什么
3. 最后强调它如何影响下一轮

### Q8. 为什么不是完全依赖 LLM 做决策？

**参考回答**

如果完全依赖 LLM，开放问题上可能灵活，但结构化诊断链会出现三个问题：

- 成本高
- 稳定性差
- 很容易做重复推理

所以我们的策略是混合式的：

- 首轮或者开放问题，LLM 负责高层推理
- 一旦已经有结构化 hint，后续轮次优先走规则和 behavior 驱动
- 如果 LLM observer 给出的 hint 不合法，还能回退到规则观察路径

这样既保留了 LLM 的推理能力，也保证了 runtime 的可控性。

**回答思路**

1. 先说全 LLM 的问题
2. 再讲混合策略
3. 最后强调“灵活性和可控性平衡”

### Q9. 你做过的关键优化是什么？

**参考回答**

一个比较关键的优化，是把多轮里的 planner 冗余调用压掉。  
之前即使 observer 已经给出了结构化 `NextHintCalls`，后续轮次也可能还会再进 `LLMPlanner`。  
后来我把逻辑改成：

- Round 1 允许 LLM 参与规划
- Round 2 以后如果已经有 `NextHintCalls`
- 就直接按 hint 续推，不再重复调用 LLM planner

这个优化把“首轮判断”和“后续机械下钻”分开了。  
对复杂诊断链来说，收益非常直接：LLM 调用次数下降，但结果链路不回退。

**回答思路**

1. 先讲原问题
2. 再讲改法
3. 最后讲收益，最好带“降调用、不回退”

### Q10. 你们怎么处理知识库不足时的外部证据补充？

**参考回答**

我们没有把外部搜索做成一条独立旁路，而是把它纳入统一的 tool workflow。  
当知识库证据不足，或者结构化查询无法直接收敛时，系统会路由到 `external_evidence_workflow`。  
这个 workflow 再内部组织：

- `web_search`
- `web_fetch`
- 结果质量和 readiness 评估

最后再把来源、可用性和质量这些信息沉淀回 tool result，并进入最终 prompt。  
这样外部证据是可追踪、可解释、可被回答消费的，而不是模型自己临时去找一些不可控的信息。

**回答思路**

1. 先讲触发条件
2. 再讲 workflow 内部步骤
3. 最后讲为什么这比裸搜索更可控

### Q11. 如果 tool 成功执行了，但最终回答还是不好，你怎么排查？

**参考回答**

我会把它当成一种“soft failure”，也就是 tool 跑通了，但语义没有被正确传递到最后答案。  
排查时我会按链路看四层：

1. `Result.Data` 有没有缺关键字段
2. `Behavior.Observe / Next` 有没有正确产出 hint 和 state
3. `RenderContext` 有没有把关键事实写进 `ToolContext`
4. `BuildGuidance` 和 prompt 里有没有正确约束答案

这类问题很多时候不是“调用失败”，而是“消费失败”。

**回答思路**

1. 先定义问题类型
2. 再按链路分层排查
3. 强调“调用成功不等于回答成功”

### Q12. 你们怎么做可观测性？

**参考回答**

Agent 系统最怕“看起来答错了，但不知道哪一步错了”，所以我会把可观测性做在每一轮。  
重点会记录：

- planning source
- 本轮调用了哪些 tool
- observer 判断是 `DONE` 还是 `CONTINUE`
- 当前轮的 `NextHintCalls`
- 是否跳过了 LLM planner

聚合层面还会看：

- round count
- tool call count
- degrade reason
- 是否生成了 `ToolContext`
- 是否生成了 `AnswerGuidance`

这样问题定位时可以区分，到底是 planner、tool、observer 还是 prompt 构造出了问题。

**回答思路**

1. 先讲为什么 agent 需要更强可观测性
2. 再讲单轮指标
3. 最后讲聚合指标和排障价值

### Q13. 如果现在继续演进这套框架，你下一步最想做什么？

**参考回答**

我不会马上推翻重做，因为当前主链路已经比较清晰。  
如果继续演进，我会优先做三件事：

1. 让首轮更多利用结构化路由，而不是默认强依赖 LLM
2. 补强 `BuildGuidance` 的生产链路，让回答约束更显式
3. 继续把集中式规则往 module behavior 下沉，减少 runtime 中央的大块语义逻辑

所以方向不是“换一套架构”，而是继续收口，让它更薄、更稳、更模块化。

**回答思路**

1. 先表达成熟判断：不推翻重来
2. 再给 2 到 3 个具体演进点
3. 用“收口”而不是“彻底重构”收尾

## 追问题

### Q14. 为什么不把所有后续决策都写在 observer 里？

**参考回答**

因为 observer 是运行时编排组件，不应该承担所有业务语义。  
如果所有续推规则都写在 observer 里，很快就会变成一个中心化的大 `switch`，每加一个新 tool 都要改中央逻辑。  
把这些语义下沉到 `Behavior`，新 tool 才能按模块扩展，observer 只负责调度和状态承接。

**回答思路**

1. 先指出中心化膨胀问题
2. 再讲模块化扩展价值
3. 最后回到 observer 的边界

### Q15. 为什么不是纯规则，为什么还要 LLM？

**参考回答**

纯规则在结构化路径上很稳定，但在开放问题、模糊表达和复杂跨域问题上会很快变硬。  
LLM 的价值在于高层理解和柔性规划，比如把用户自然语言问题映射成更合适的第一跳工具。  
所以我们的架构不是“规则替代 LLM”，而是“规则和 LLM 各司其职”。

**回答思路**

1. 先讲纯规则的短板
2. 再讲 LLM 更擅长哪里
3. 最后强调协同而不是替代

### Q16. 如果多个 tool 给出了冲突证据，你们怎么处理？

**参考回答**

我会优先相信更深层、更实时、更接近一手事实的证据。  
比如文档级诊断说失败，但更后面的 task/node 级状态显示还在 running，那最终回答应该以 task/node 的实时状态为准，同时显式说明前后证据存在冲突。  
也就是说，系统不能只是拼接结果，而要在 guidance 或回答逻辑里处理证据层级。

**回答思路**

1. 先给出证据优先级原则
2. 再举一个冲突例子
3. 最后强调“显式说明冲突”

### Q17. 如果新增一个工具族，比如 GitHub 或数据库查询，你怎么接入？

**参考回答**

我会按现有 module-first 方式接入，而不是绕过框架直接写特例。  
通常会做这几步：

1. 先定义 invoker 和输入输出结构
2. 补 spec，明确能力域、风险、证据来源
3. 补 behavior，定义结果续推和上下文消费方式
4. 在 assembly 层注册
5. 补齐运行时和回归测试

这样接入的不是“一个能跑的工具”，而是“一个能被 agent loop 稳定消费的模块”。

**回答思路**

1. 先强调“按框架接入”
2. 再按落地步骤讲
3. 最后强调不是只写 invoker

## 回答思路卡片

这一部分专门保留“作答方法”，方便你脱稿发挥。

### 卡片 1：定义类问题

适用：

- 为什么要做这套框架？
- 核心抽象是什么？
- 为什么需要 behavior？

答题骨架：

1. 背景问题
2. 原方案不够
3. 你的抽象解决了什么
4. 一句话总结价值

### 卡片 2：链路类问题

适用：

- 一个 tool 怎么走到最终回答？
- Plan-Act-Observe 怎么串起来？

答题骨架：

1. 选一个具体例子
2. 按时间顺序讲
3. 每步只说输入、处理、输出
4. 最后说明这条链解决了什么

### 卡片 3：优化类问题

适用：

- 你做过什么关键优化？
- 为什么这么改？

答题骨架：

1. 优化前的问题
2. 关键改动
3. 影响到的主链路
4. 收益和边界

### 卡片 4：排障类问题

适用：

- 为什么 tool 跑了答案还是差？
- 你怎么 debug agent runtime？

答题骨架：

1. 先区分硬错误和软退化
2. 再按链路逐层排查
3. 优先看数据契约、状态流、context、prompt
4. 最后补可观测性

### 卡片 5：取舍类问题

适用：

- 为什么不是全 LLM？
- 为什么不是纯规则？
- 为什么是 module-first？

答题骨架：

1. 先讲两个极端方案的缺点
2. 再讲你们为什么选中间方案
3. 讲清职责边界
4. 最后落到可控性和扩展性
