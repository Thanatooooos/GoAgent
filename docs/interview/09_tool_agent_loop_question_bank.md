# 09 Tool 与 Agent Loop 高质量问题清单

## 说明

这份文档保留更接近真实技术面试的问题质量和问题组织方式。

和 `07_tool_agent_loop_interview.md`、`08_tool_agent_loop_interview_qa.md` 的定位不同：

- `07` 更像复习提纲
- `08` 更像模拟问答稿
- `09` 更像面试官问题库

如果你的目标是准备深入技术面，应该优先看这份；如果你的目标是练口述表达，再配合 `08` 使用。

---

## 架构理解

1. 你说“统一注册、执行与结果消费流程”，这里你统一的最小抽象是什么？为什么这样拆？
2. 一个 tool 从定义到运行时被消费，完整链路是怎样的？
3. 为什么要把“执行工具”和“理解工具结果”拆开？如果不拆会出什么问题？
4. `Tool / ToolModule / Registry / Executor / Behavior / AgentLoop` 这些角色各自负责什么？
5. 你们为什么需要 `Behavior` 这一层？它解决了什么仅靠 tool 本身解决不了的问题？
6. `Spec` 这种元数据层的价值是什么？它除了描述工具还能参与什么运行时决策？

---

## 运行时机制

1. 你们的 `Plan-Act-Observe` 多轮机制是怎么运转的？每一轮输入输出分别是什么？
2. `planner`、`executor`、`observer` 之间是怎么传递状态的？
3. 什么情况下会继续下一轮，什么情况下会停止？
4. 你们怎么表示“下一步建议调用哪个 tool”？为什么用这个表示方式？
5. 为什么需要 `AgentState`？它和单轮 `Result` 的关系是什么？
6. 多轮链路里，前一轮结果如何影响下一轮 planning？

---

## 结果消费与回答生成

1. tool 的执行结果最后是怎么进入大模型上下文的？
2. `RenderContext` 和 `BuildGuidance` 的区别是什么？为什么要分成两条通道？
3. 如果一个 tool 能查到数据，但回答质量仍然差，你会优先检查哪几个环节？
4. 你们如何避免模型只看到 raw result，却无法正确理解其业务语义？

---

## 设计取舍

1. 为什么不是把所有“下一步决策”都交给 LLM，而是保留规则和 behavior 驱动？
2. 规则路径和 LLM 路径各自适合什么场景？你们怎么做 fallback？
3. 为什么要支持 `module-first`，而不是只保留一个简单的 tool function registry？
4. 当 tool 越来越多时，如何避免中心化 `switch/case` 膨胀？
5. 如果新加一个工具族，比如 GitHub 或 DB 查询，你会如何接入这套框架？

---

## 复杂场景与失败处理

1. 如果 planner 给出的 tool 不存在，或者参数非法，系统怎么处理？
2. 如果 LLM observer 产生了幻觉 hint，你们怎么防止它把链路带偏？
3. 多轮循环最容易出现哪些退化问题？你们怎么监控和发现？
4. 如果一个 tool 成功执行了，但没有触发后续诊断，你怎么排查？
5. 外部证据检索质量不稳定时，你们如何决定“继续搜”还是“现在就回答”？
6. 如果多个 tool 返回相互冲突的证据，最终答案以谁为准？

---

## 性能与稳定性

1. 多轮 `AgentLoop` 的主要性能开销在哪？你们做过哪些优化？
2. 你们如何减少冗余的 LLM 调用？
3. 并行 tool call 是怎么设计的？什么场景能并行，什么场景必须串行？
4. `MaxIterations` 这种限制为什么需要？它的副作用是什么？
5. 如果线上复杂问题常常在 3 轮内收敛不了，你会怎么改，而不是简单调大轮数？

---

## 可观测性与验证

1. 你们如何知道一个问题到底是 planner 选错了、tool 执行错了，还是 observer 判断错了？
2. 这套框架你会打哪些关键日志或 trace 字段？
3. 你如何验证“优化后真的减少了 LLM 调用，但答案没有变差”？
4. 单元测试和集成测试分别该覆盖哪些点？
5. 你们怎么构造回归样例去覆盖 `doc/task/trace/external evidence` 这几类链路？

---

## 落地效果与复盘

1. 这套框架上线前后，最明显的收益指标是什么？
2. 你做过的最关键一次架构收口或重构是什么？为什么当时必须做？
3. 如果现在让你重做一遍，这套设计你最想改的地方是什么？

---

## 我最看重的 5 个高频追问

1. 为什么需要 `Behavior`，不用它会怎样？
2. 一个具体 tool 从注册到最终回答，完整调用链是什么？
3. 多轮决策里，状态是怎么流动和收敛的？
4. 规则和 LLM 各自负责什么，为什么不能只保留一边？
5. 你们怎么证明这不是“能跑”，而是真的“可维护、可扩展、可观测”？

---

## 使用建议

建议这样配合使用：

1. 先用这份文档挑 8 到 10 个高频问题，确认问题树是否覆盖完整。
2. 再去看 `08_tool_agent_loop_interview_qa.md`，把这些问题转成你自己的回答版本。
3. 最后自己再做一轮“脱稿回答”，看能不能只靠问题把链路讲顺。

如果只看答题稿，很容易讲得顺但追问一深就散；如果先用这份问题库搭骨架，再回到答题稿，你的面试稳定性会好很多。
