# Agent 能力扩展建设方案

日期：2026-05-12

状态：Draft v1

适用范围：当前 `internal/app/rag/tool` Agent 体系在未来 1-3 个迭代内的能力建设、架构收口与风险控制。

## 一、文档目的

当前 Agent 已具备基础多轮工具编排能力，并完成了诊断链路、RAG 融合、联网搜索补充、trace 与 SSE 基础可观测性建设。下一阶段的核心问题不再是“能不能加一个新 tool”，而是“如何在不把 Agent 变成高耦合规则网络的前提下，持续增加新的能力域”。

本方案聚焦以下目标：

1. 明确当前 Agent 的健康度、边界和真正的扩展瓶颈。
2. 给出未来新增能力的统一扩展模式，尤其覆盖：
   - 自动搜索
   - 信源判断
   - 自动获取文档
   - 自动入库
3. 规划未来 1-3 个迭代的实施顺序、阶段目标、风险闸门与验收标准。

本方案不替代以下文档：

1. [agent_state_assessment.md](/abs/path/D:/goagent/docs/agent_state_assessment.md)
   用于描述当前 Agent 的状态评估。
2. [agent_evolution_plan.md](/abs/path/D:/goagent/docs/agent_evolution_plan.md)
   用于描述 Agent 自主化方向与决策能力演进。
3. [agentic_chat_dev_plan.md](/abs/path/D:/goagent/docs/agentic_chat_dev_plan.md)
   用于描述 Agentic Chat 的早期开发方案与已完成范围。

本方案重点补充“从现在开始怎么做”。

## 二、当前基线判断

### 2.1 架构健康度

当前 Agent 架构整体健康，具备继续建设的基础，主要体现在：

1. 主流程已经闭环。
   当前已形成 `rewrite/retrieve -> AgentLoop -> tool execution -> observe -> answer rendering` 的完整链路。
2. 分层方向正确。
   `AgentLoop` 负责编排，`Planner/Observer` 负责决策，`Registry/Executor` 负责执行，Graph Tool 承接确定性工作流。
3. 新能力已经证明可接入。
   `web_search`、`web_fetch`、诊断 graph tool 的接入说明当前架构支持横向扩展。
4. 可观测性基础已经具备。
   目前已有 trace、answer guidance、SSE 事件和测试回归能力。

### 2.2 当前扩展瓶颈

真正的瓶颈已经从“工具数量”转移到了“决策与契约维护成本”，主要体现在：

1. Agent 编排层偏重。
   `AgentLoop` 仍承担较多职责，包括入口路由、结果驱动推进、并发执行、状态归并和渲染协作。
2. 决策边界容易模糊。
   `base rules`、`Planner`、`Observer`、`nextAction`、`answer guidance` 都在影响流程行为，新增场景时容易跨多处联动。
3. 结果契约偏动态。
   `Result.Data` 目前足够灵活，但字段契约分散在多个消费者中，维护者需要记住大量隐式约定。
4. 写操作安全机制尚未成型。
   当前 Agent 更适合只读查询和诊断；一旦进入抓取、入库、提交任务等动作，风险等级会显著上升。

### 2.3 总体结论

当前 Agent 适合继续扩展，但扩展方式必须从“继续加 tool”升级为“按能力域建设 capability pipeline”。否则短期虽然能跑，长期会出现以下问题：

1. 场景越多，入口规则越乱。
2. 中间链路越多，局部推进规则越重。
3. 输出越复杂，证据组织和可信度判断越难统一。
4. 一旦涉及写操作，没有明确闸门会放大误操作风险。

## 三、未来建设原则

### 3.1 工具是原子动作，能力是业务单元

未来不应把“自动搜索”“自动入库”理解成单个 tool，而应把它们建设成能力流水线：

1. Tool
   原子动作，例如 search、fetch、parse、submit。
2. Capability
   面向业务目标的固定链路，例如“搜索并判断信源”“发现文档并给出入库计划”。
3. Agent
   负责编排能力、处理多轮上下文、决定是否继续或终止。

结论：后续新增复杂场景时，优先新增 capability，而不是直接增加离散 tool 数量。

### 3.2 读写分离

未来能力按风险等级分成两类：

1. 只读能力
   搜索、抓取、判断、总结、诊断、验证。
2. 写能力
   触发入库、更新状态、创建任务、写知识库。

写能力默认不得走“无确认自动执行”路径。未来所有自动入库类能力都应至少分为两阶段：

1. `prepare`
   发现候选、解析元数据、生成计划。
2. `commit`
   显式提交，且保留 trace 和来源依据。

### 3.3 证据优先于结论

未来新增能力必须输出结构化证据，不允许只产出自然语言结论。至少需要保证：

1. 搜索结果有来源与标题。
2. 抓取结果有 URL、正文摘要、截断标记。
3. 信源判断有评分依据。
4. 入库建议有来源、目标库、策略和风险提示。

这样做的目的不是为了“展示更多细节”，而是为了后续：

1. Observer 可判断是否继续。
2. 回答渲染可引用证据。
3. trace 可复盘。
4. 自动化评测可落地。

### 3.4 决策边界要持续守住

未来建设过程中，要持续守住以下边界：

1. `Planner / base rules`
   决定入口路线和能力域。
2. `Capability workflow / graph`
   负责确定性中间流程。
3. `nextAction` 或后续 transition policy
   只负责单结果驱动的局部推进。
4. `Observer`
   决定停止、继续、降级、补充搜索。
5. `Answer guidance / renderer`
   负责最终表达，不负责重新规划主流程。

只要这条边界被打破，系统就会重新回到“规则分散、多点互相覆盖”的状态。

## 四、目标能力图谱

### 4.1 能力域划分

建议未来能力域按以下方式分层：

1. `diagnosis`
   面向导入失败、节点异常、任务状态排障。
2. `knowledge`
   面向 RAG 问答和知识库内证据组织。
3. `search`
   面向知识库外补充搜索、抓取和摘要。
4. `source_evaluation`
   面向外部来源可信度、时效性和引用风险判断。
5. `document_acquisition`
   面向文档发现、抓取、解析、标准化。
6. `ingestion`
   面向候选入库、计划生成、提交执行和结果验证。

### 4.2 建议的能力流水线

#### A. 自动搜索能力

建议链路：

`query rewrite -> web_search -> result dedup -> web_fetch -> content extract -> source evaluate -> answer guidance`

用途：

1. 补足知识库证据不足时的外部来源。
2. 处理“最新”“官网说明”“外部产品文档”等问题。
3. 作为文档发现能力的前置输入。

#### B. 信源判断能力

建议链路：

`source classify -> freshness inspect -> corroboration inspect -> trust score -> risk hint`

建议输出字段：

1. `sourceType`
2. `trustScore`
3. `freshness`
4. `corroboration`
5. `riskHints`
6. `citationRecommendation`

用途：

1. 控制回答引用优先级。
2. 为自动抓取和自动入库提供前置筛选。
3. 在搜索结果多、质量参差不齐时限制幻觉性引用。

#### C. 自动获取文档能力

建议拆成两个子能力，而不是一个大 tool：

1. `document_discovery`
   负责发现 URL、站点目录、文档入口、sitemap、API 端点。
2. `document_fetch_and_parse`
   负责下载、正文提取、格式识别、标准化、切分前清洗。

用途：

1. 为知识库补充文档来源。
2. 为自动入库生成标准化输入。
3. 为“发现但暂不入库”的模式保留只读链路。

#### D. 自动入库能力

建议链路：

`candidate prepare -> ingestion plan -> policy check -> human confirm or guarded auto commit -> task verify`

建议拆分工具职责：

1. `ingestion_candidate_prepare`
2. `ingestion_plan`
3. `ingestion_submit`
4. `ingestion_verify`

用途：

1. 避免“搜索到内容就直接写库”。
2. 把入库前判断、计划与提交动作解耦。
3. 让系统支持半自动和全自动两种模式。

## 五、建议的架构扩展方式

### 5.1 从 Tool Set 升级为 Capability Set

现阶段 Agent 主要围绕“有哪些 tool 可调用”组织。下一阶段建议升级为“两层组织”：

1. Core Tool Layer
   提供原子动作能力。
2. Capability Layer
   提供固定业务流水线和领域策略。

推荐职责划分如下：

1. `tool/core`
   `Tool`、`Call`、`Result`、`Registry`、`Executor`、通用 helper。
2. `tool/agent`
   编排入口、回合控制、上下文渲染、状态收敛。
3. `tool/observer`
   停止条件、证据校验、降级与继续策略。
4. `tool/graph`
   确定性流程图工具。
5. `capability/search`
   search/fetch/extract/dedup/citation。
6. `capability/source`
   信源评分与来源分类。
7. `capability/document`
   discovery/fetch/parse/normalize。
8. `capability/ingestion`
   candidate/plan/submit/verify。

注：当前不要求立刻引入 `capability/` 目录，但后续代码建设应按此边界设计接口。

### 5.2 结果契约逐步视图化

不建议现在立刻彻底放弃 `Result.Data map[string]any`，但建议为关键能力增加 typed view：

1. `WebSearchResultView`
2. `WebFetchResultView`
3. `SourceEvaluationResultView`
4. `DocumentAcquisitionResultView`
5. `IngestionPlanResultView`

收益：

1. Observer 不再直接读散落字段。
2. Renderer 不再每次重新猜字段结构。
3. 新人阅读成本显著下降。
4. 自动化测试更稳定。

### 5.3 局部状态推进机制维持轻量

当前 `nextAction` 仍有价值，但要明确它的适用边界：

1. 适合处理单结果驱动的下一跳。
2. 不适合承载入口规划或全局能力选择。
3. 不适合变成多能力混合状态机。

建议做法：

1. 近期保留现状。
2. 当链路型 tool 超过当前规模后，再升级为 `transition registry`。
3. 升级时按结果类型注册 handler，不做一个巨型中心 switch。

### 5.4 写操作增加统一闸门

自动入库、提交任务、写状态等能力必须经过统一策略层，至少校验：

1. 来源是否可信。
2. 内容是否完整。
3. 是否命中允许自动写入的场景。
4. 是否存在重复文档或覆盖风险。
5. 是否需要人工确认。

建议在能力层引入统一概念：

1. `ExecutionMode`
   `read_only` / `proposal_only` / `guarded_write`
2. `RiskLevel`
   `low` / `medium` / `high`
3. `ApprovalRequirement`
   `none` / `recommended` / `required`

## 六、分阶段实施计划

### Phase 0：架构收口与基础设施补强

目标：在不扩大能力面的前提下，为后续扩展降低耦合风险。

建议内容：

1. 继续压缩 `AgentLoop` 职责。
2. 把关键结果逐步补充 view 层。
3. 固化搜索相关结果字段契约。
4. 把只读能力和写能力的执行模式在上下文里显式化。
5. 完善 capability 级 trace 结构。

验收标准：

1. 新增一个搜索链路中间 tool，不需要跨 4 个以上位置改逻辑。
2. trace 能清晰还原“为什么触发搜索”“为什么停止”“为什么不给写入”。
3. 回答渲染能区分 KB 证据和外部证据。

### Phase 1：搜索与信源评估能力成型

目标：先做高价值、低风险的只读能力域。

建议内容：

1. 完成搜索能力流水线闭环。
2. 增加结果去重和正文抽取质量控制。
3. 引入信源评分结构。
4. 在回答阶段按信源等级排序引用。
5. 建立一批“KB 不足 -> 搜索补充 -> 回答”的评测样例。

验收标准：

1. Agent 能稳定区分“无需搜索”和“应搜索”。
2. 引用外部结果时能说明来源类型。
3. 多结果场景下不会优先引用明显低可信内容。

### Phase 2：文档获取能力建设

目标：把“搜索到网页”提升为“发现并提取文档”。

建议内容：

1. 建立文档发现能力。
2. 支持多种来源适配器。
3. 建立标准化解析输出。
4. 支持“只发现不入库”的中间态。

验收标准：

1. 给定一个站点或主题，系统能输出可入库候选列表。
2. 文档抓取结果具备来源、摘要、内容完整度和解析状态。
3. 失败路径可追踪，不吞错。

### Phase 3：入库计划能力建设

目标：让 Agent 具备“提出入库方案”的能力，但默认不自动写库。

建议内容：

1. 基于文档候选生成入库计划。
2. 补充目标知识库选择逻辑。
3. 补充 chunk 策略、标签、覆盖风险判断。
4. 输出结构化计划，而不是直接提交。

验收标准：

1. Agent 能解释“为什么建议入这个库”。
2. Agent 能说明是否存在重复或覆盖风险。
3. Agent 能给出明确的提交前检查项。

### Phase 4：受控写入与验证闭环

目标：在明确闸门和审计机制后，开放有限自动写入。

建议内容：

1. 为 `ingestion_submit` 增加策略校验。
2. 提交后自动跟踪任务状态。
3. 在失败时给出自动诊断或人工介入建议。
4. 所有写入动作进入 trace 和审计日志。

验收标准：

1. 系统不会在来源不可信或内容不完整时自动写库。
2. 所有写操作都可追溯到来源、计划、风险判断和提交结果。
3. 提交失败时能够给出明确回滚或补救建议。

## 七、风险评估

### 7.1 架构风险

风险：

1. 继续只按 tool 累加，最终形成大而散的规则网。
2. 决策职责跨 Planner、Observer、Renderer 反复覆盖。

控制措施：

1. 以 capability 为建设单元。
2. 重要决策边界写入代码约束和文档约束。

### 7.2 可信度风险

风险：

1. 外部搜索结果引用不稳定。
2. 内容新旧冲突或转载失真。

控制措施：

1. 引入信源评分。
2. 区分官方、厂商、社区、聚合转载。
3. 支持多来源交叉印证。

### 7.3 数据安全风险

风险：

1. 自动抓取到不适合入库的内容。
2. 自动写入导致重复、污染或错误覆盖。

控制措施：

1. 读写分离。
2. 写前策略闸门。
3. 提交前保留 proposal 阶段。

### 7.4 运维风险

风险：

1. 外部抓取链路失败率高。
2. 不同来源格式差异大。
3. 网络能力不可用时链路退化。

控制措施：

1. 适配器分层。
2. 失败分类与重试策略。
3. 明确“无网络/抓取失败”时的降级回答。

### 7.5 评测风险

风险：

1. 功能虽然能跑，但无法稳定验证质量。

控制措施：

1. 建立 capability 级场景集。
2. 对搜索、信源、抓取、入库计划分别设计回放样例。
3. 对关键策略建立回归测试和 trace 样本。

## 八、建议的评测与验收体系

### 8.1 测试分层

建议至少分为四层：

1. 单元测试
   工具输入输出、结果 view、策略判断。
2. 工作流测试
   graph/capability 内部链路推进。
3. Agent 集成测试
   多轮 `plan -> act -> observe` 行为验证。
4. 场景回放评测
   真实问题样本、固定预期行为、trace 快照。

### 8.2 重点样例池

建议至少覆盖以下样例：

1. KB 证据充足，不触发外部搜索。
2. KB 不足，触发搜索并抓取外部内容。
3. 搜索结果来源多样，能优先选择高可信来源。
4. 发现文档但不满足入库条件，只生成 proposal。
5. 入库计划命中重复风险，拒绝自动提交。
6. 自动提交后任务失败，能进入诊断或给出补救建议。

### 8.3 关键指标

建议跟踪以下指标：

1. 搜索触发正确率
2. 来源可信度排序正确率
3. 文档抓取成功率
4. 解析有效内容比例
5. 入库计划准确率
6. 自动写入误触发率
7. trace 可复盘完整率

## 九、优先级建议

如果从现在开始立刻做，建议按以下顺序推进：

1. 先做搜索能力和信源评估能力
   价值高、风险低、可以立刻改善 Agent 回答质量。
2. 再做文档获取能力
   为后续知识库补充和自动入库打基础。
3. 然后做入库计划能力
   先输出方案，不急于自动提交。
4. 最后做受控写入
   只有在来源可信、计划可解释、验证链路完整后再开放。

不建议当前阶段优先做的事情：

1. 不建议立即做完全自动写库。
2. 不建议为了“更聪明”过早引入多 Agent 拆分。
3. 不建议把所有新场景都塞进统一 prompt 规则里。

## 十、实施结论

从当前状态出发，Agent 的下一阶段建设应该从“增量加 tool”转向“按能力域扩展 capability pipeline”。其中最值得率先投入的是搜索、信源判断和文档获取三类只读能力；自动入库必须采用 proposal-first、guarded-write 的建设方式。

如果按本方案推进，预期收益包括：

1. Agent 回答质量提高，尤其在 KB 不足场景。
2. 外部引用更加可解释、可追溯。
3. 文档补充和入库流程逐步自动化，但不会过早失控。
4. 后续新增能力时，扩展点更清晰，架构债务增长速度更可控。

简化表述如下：
