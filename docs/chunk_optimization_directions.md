# Chunk Optimization Directions

更新时间：2026-04-30

这份文档用于整理 `knowledge` 模块后续的 chunk 优化方向，目标是让当前切分能力从“工程可用”逐步提升到“更接近语义级别分割”。

## 当前现状

当前系统已具备两类 chunk strategy：

- `fixed_size`
- `markdown`

其中 `markdown` 已支持：

- 基于标题块分段
- 按空行切分基本段落
- 超长块退回 fixed size 切分

但当前方案仍然偏“结构感知初级版”，距离更稳定的语义级分块还有明显差距。

## 优化目标

后续 chunk 优化建议围绕以下四个目标展开：

1. 切分边界尽量贴近语义单元，而不是字符长度
2. 检索召回时既要命中精确，又要保留足够上下文
3. 为后续 rerank、重组、溯源补足 metadata
4. 建立面向真实问答效果的评测基线

## 优先优化方向

### 1. Markdown 分层切分增强

这是最优先的方向，收益最高、与当前实现最连续。

建议增强点：

- 识别标题层级，而不只是“遇到标题就切块”
- 将 `h1 / h2 / h3` 的路径保留到 metadata
- 避免不同小节内容混入同一块
- 对列表、引用、表格、代码块做单独处理
- 保证代码块、表格等结构不被普通段落逻辑打散

预期收益：

- 块边界更接近文档作者原始结构
- 检索命中后更容易拿到完整的小节上下文
- 适合技术文档、接口文档、说明文档

### 2. 基于句段边界的动态切分

当前超长块退回 fixed size，容易把句子或语义单元切断。

建议增强点：

- 优先按段落、句号、列表项等自然边界切分
- 在接近目标长度时优先合并相邻短段
- 仅在找不到合适边界时再硬切
- 减少无意义的字符级切断

预期收益：

- 显著减少“半句被截断”的情况
- 普通说明文、FAQ、纯文本材料的切分质量会更稳定

### 3. Chunk Metadata 增强

即使正文切分逻辑暂时不大改，metadata 增强也能明显提升后续检索和拼装质量。

建议至少补充：

- 标题路径
- 块类型
  - `heading`
  - `paragraph`
  - `list`
  - `table`
  - `code`
- 原文顺序范围
- 父 section id
- 原始文档局部位置索引

预期收益：

- 方便后续做 rerank
- 方便前端展示命中片段上下文
- 方便做“同 section 邻接块补全”

### 4. 父子块双层结构

这是中高收益方向，适合在基础 chunk 质量稳定后推进。

思路：

- 先生成较大的“父块”
- 再在父块内部派生较小的“检索子块”

建议方式：

- 检索阶段命中子块
- 拼接上下文阶段回带父块或相邻子块

预期收益：

- 子块更利于精确召回
- 父块能保留上下文完整性
- 对长文档问答会有明显帮助

## 次级优化方向

### 5. 语义 Overlap 替代固定 Overlap

当前 overlap 更偏字符级。

可以考虑：

- 保留上一块最后 1 到 2 个句子
- 或保留当前 section 标题
- 对代码块、表格使用更保守的 overlap 规则

相比固定 overlap，这种方式更自然，也更节省 token。

### 6. 按文档类型分派不同 Chunker

后续不应只区分 `fixed_size` 和 `markdown`。

可以逐步扩展为：

- Markdown 文档：结构感知 chunker
- 纯文本：句段边界 chunker
- FAQ/制度文档：问答对或条款级 chunker
- 代码说明文档：按类、函数、代码块切分

### 7. 面向检索效果的参数调优

chunk 优化不能只看“切得是否好看”，更要看真实召回效果。

建议后续建立评测维度：

- 命中率
- 上下文完整性
- 冗余率
- 回答准确性
- token 成本

## 推荐落地顺序

建议按以下顺序推进：

1. Markdown 分层切分增强
2. 句段边界动态切分
3. Chunk Metadata 增强
4. 父子块双层结构
5. 语义 Overlap
6. 多文档类型专用 Chunker
7. 检索评测与参数调优

## 第一阶段建议

如果只做一轮投入最小、收益最大的优化，建议先完成：

### 第一批

- `structure_aware / markdown` 语义统一
- Markdown 标题层级切分
- 标题路径 metadata
- 列表 / 代码块基础识别

### 第二批

- 段落与句段边界动态合并
- 长块切分时优先按自然边界而不是直接 fixed size

### 第三批

- 父子块模型
- retrieval 质量评测样本

## 近期不建议优先投入的方向

以下方向可以暂缓，不建议先做：

- 复杂可视化配置界面
- 过早引入太多 chunk strategy 名称
- 在没有评测基线前做大规模参数微调

当前更重要的是先把最常见的 Markdown / 文本型知识文档切好、切稳、切得更接近语义边界。
