# Memory Architecture Design

讨论日期：2026-05-17

这份文档记录了关于对话记忆分层、长消息处理和长期记忆入库策略的设计讨论与阶段性结论。

## 背景

当前项目已有短期记忆能力（conversation_message + memory compression），但缺少：
1. 对超长单条消息的处理策略
2. 短期/长期记忆的显式分层
3. "哪些消息值得跨对话记住"的识别机制

## 1. 两层记忆边界

```
短期记忆                          长期记忆
──────────────────────────────────────────────────
生命周期：当前对话窗口内           生命周期：跨对话持久化
存储方式：conversation_message    存储方式：knowledge document + vector
检索方式：直接塞上下文             检索方式：semantic retrieve
内容：本轮问题、上下文、工具结果   内容：用户偏好、领域知识、反馈模式
```

短期记忆已有：`LoadHistory` + `CompressIfNeeded`（多轮压缩）。长期记忆需要新建。

## 2. 长消息处理

### 核心结论：写时摘要 > 读时摘要

超长消息内容固定，摘要只需回答"这条消息讲了什么"，不需要根据后续 query 调整。写时生成一次，永久复用。

### 摘要生成策略

- **规则优先**：日志/代码/结构化内容用规则提取（零延迟）
- **LLM 补充**：通用文本走 LLM 摘要（高质量但有延迟）
- **并存**：摘要写入 `message.Content`，原文存入 `ThinkingContent` 或独立 `LongContent` 字段

### 接入点

`ConversationMessageService.AddMessage()` —— 所有消息的必经入库点，在写入前做检测和摘要替换。

## 3. 消息分类：哪些值得入库

### 明确该入库（高信号）
- 用户显式偏好声明："以后都用中文回答"、"习惯先看代码再看文档"
- 用户纠正系统行为后的反馈模式
- 用户提供的领域知识："我们项目用了自定义 chunker"
- Agent 成功路径被用户确认

### 明确不该入库（噪音）
- 问候、闲聊、一次性问题
- 日志/错误堆栈原文（即使有价值也是"文档"而非"记忆"）
- 中间工具调用的结果摘要
- 所有 assistant 回答（除非被用户明确肯定）

### 灰色地带
- 用户描述的复杂问题场景（隐含领域上下文）
- 用户隐含偏好（连续拒绝某种策略）

## 4. 识别策略：多层漏斗

```
第一层：用户显式标记（零误判，高精度）
  → /remember 命令、点赞回答、手动保存
  → 立即可做

第二层：写入时 LLM 分类（异步，不阻塞请求）
  → 消息入库后，异步调 cheap model 判断分类
  → 维度：偏好声明 / 领域知识 / 反馈模式 / 纯临时

第三层：会话结束时聚合推断（全局视角）
  → 整场对话结束后回顾，提取：
    - 用户偏好变化
    - 被反复提及的实体/概念
    - 被纠正过的错误模式
```

## 5. 长期记忆入库形态

复用现有 ingestion → knowledge document 链路：

```
识别出值得记忆的消息
  → 创建 knowledge document（sourceType=memory）
  → pipeline: parser → chunker → enricher → indexer → vector store
  → 标签: memory_type=preference|knowledge|feedback
  → 后续对话通过 retrieve 自动召回
```

### 待决策：全局 vs 按知识库隔离

- 方案 A：所有对话记忆放进一个专门的 "user_memory" 知识库，全局共享
- 方案 B：对话绑定到哪个知识库，记忆归哪个知识库
- **倾向：全局 + 按知识库标签过滤**——偏好是全局的，领域知识可能是特定库的

## 6. 长消息处理与长期记忆的关系

两者是独立但并行的处理链路：

```
超长消息
  ├── 规则/LLM 摘要 → message.Content（短期记忆，塞上下文）
  └── 记忆提取 → knowledge document（长期记忆，入库 + 向量化）
```

## 7. 当前状态与下一步

### 已达成共识
- 两层记忆边界清晰：短期（conversation_message）vs 长期（knowledge document + vector）
- 长消息摘要写时生成，规则优先 + LLM 补充
- 长期记忆复用已有 ingestion pipeline
- 识别策略分三层漏斗，第一层可立即落地

### 待进一步讨论
- 长期记忆是全局 user_memory 知识库还是按项目隔离
- 第二层/第三层 LLM 分类的 prompt 设计
- 记忆的更新/过期/冲突策略
- 与已有 knowledge base 权限模型的关系
