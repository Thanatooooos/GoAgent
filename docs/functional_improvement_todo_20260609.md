# GoAgent 功能完备性改进清单（2026-06-09）

## 背景

这份清单基于当前仓库实现和面试追问角度整理，重点不是代码风格或架构洁癖，而是：

- 从功能层面看，哪些能力已经有 `v1 能跑` 的闭环
- 哪些地方还不够完备，容易在技术面中被追问穿
- 后续应该按什么优先级补强，才能让能力从“可用”走向“可证明、可治理、可统一”

目标是把当前项目中最容易被质疑的能力收敛成一组明确的改进任务。

---

## 总体判断

当前项目的主要问题不是“没做功能”，而是很多能力停留在：

- 有主链路
- 有模块实现
- 能在典型场景跑通

但还没有完全做到：

- 可治理
- 可评估
- 可解释
- 可统一

也就是说，当前更像一组已经跑起来的 `v1` 能力，而不是全部收口成成熟的平台能力。

---

## P0：必须优先补强

### 1. 收口 Query Rewrite 与术语治理边界

#### 当前问题

- 当前 `rewrite` 已支持 `RewrittenQuestion / SubQuestions / NeedRetrieval`
- 但历史上更多是 LLM 改写，术语治理能力较弱
- 即使现在补了轻量词典规则，也仍然不是完整的术语管理体系

#### 改进任务

- 建立术语词典维护机制
- 支持高频别名命中统计
- 支持未命中别名回流
- 支持词条版本化和配置审计
- 明确 LLM 改写和术语归一化的职责边界

#### 验收标准

- 能明确回答“术语词典谁维护、怎么更新、如何发布”
- 能统计常见 alias 命中率
- 能追踪高频未覆盖术语
- 面试时不再只能说“交给 LLM 处理”

### 2. 为 Rewrite 增加强约束校验与回退规则

#### 当前问题

- 当前兜底更多是“失败时回退原问题”
- 对关键约束丢失的检测还不够强
- LLM 输出不稳定时，可控性主要依赖 fallback，而不是前置约束

#### 改进任务

- 增加关键字段保留校验
- 覆盖服务名、错误码、时间范围、资源名等约束项
- 对 rewrite 和 split 结果增加结构化检查
- 不通过时回退原 query，并记录原因

#### 验收标准

- 能列出明确的 rewrite 回退条件
- 能给出真实案例说明“什么情况下会拒绝模型改写结果”
- trace 中可观测回退原因

### 3. 补齐 Retrieve 评估闭环

#### 当前问题

- 已有 retrieve 评估工具
- 但更偏检索层指标
- “召回率提升”容易停留在主观结论

#### 改进任务

- 构建人工标注评测集
- 区分问题类型：口语化、缩写、跨轮指代、多条件问题
- 固定 baseline，对比 rewrite/split 前后的效果
- 固化 `Recall@K / Hit@K / MRR`

#### 验收标准

- 有稳定评测集和统计流程
- 可以回答“提升的是哪类问题，不是泛化地说所有问题都提升”
- 评测可以重复运行，不依赖临时人工观察

### 4. 收口双 Agent Runtime

#### 当前问题

- 当前 `RagChatService.Chat(...)` 仍存在新旧两套路径
- 业务入口依赖 `UseAgentRuntime` 选择不同执行链路
- “统一工具框架”和“统一 agent runtime”还没有真正落地成单主线

#### 改进任务

- 明确新旧 runtime 的主次关系
- 给出旧 `rag/tool` runtime 的退役计划
- 统一输出契约和主链路观测点
- 尽量减少业务层 boolean 决定走哪条大链路

#### 验收标准

- 能说清楚哪套 runtime 是默认主路径
- 新 capability 不需要再考虑是否接两套系统
- 面试时不会被问出“到底哪套才是生产主线”

---

## P1：本阶段应该完成

### 5. 把会话窗口控制从消息数升级为 Token Budget

#### 当前问题

- 当前上下文窗口更接近按最近 N 条消息控制
- 对长消息场景不够精细
- `HistoryKeepTurns` 的命名和真实语义容易混淆

#### 改进任务

- 增加 token-aware 上下文预算控制
- 区分“消息条数上限”和“prompt token 上限”
- 优先按 token 裁剪，再按轮次或消息数兜底

#### 验收标准

- 能准确说明上下文裁剪依据是 token，而不是模糊的“最近 N 轮”
- 长消息场景不会因少量超长消息导致上下文失控

### 6. 将摘要压缩改为异步

#### 当前问题

- 当前摘要压缩挂在写入路径上
- 会增加响应时延
- 更像功能优先的同步实现

#### 改进任务

- 将摘要压缩改为异步任务
- 允许后台延迟生成和持久化 summary
- 避免主请求首包时延被摘要任务拖慢

#### 验收标准

- 首包时延和总响应时延有压缩前后对比
- 主请求不会因为 summary 生成而明显阻塞

### 7. 增强 Summary 质量控制

#### 当前问题

- 当前 summary 结构较轻
- 失真风险主要靠“摘要 + 最近窗口消息”缓解
- 缺少强校验和回滚机制

#### 改进任务

- 增加摘要质量校验
- 保证关键实体、关键结论不丢
- 增加 summary 异常重建或重压缩机制
- 评估是否引入摘要质量标记

#### 验收标准

- 能回答“摘要错了怎么发现、怎么止损”
- 长会话漂移问题可以被明确观测到

### 8. 升级 `t_conversation_summary` 设计

#### 当前问题

- 当前 summary 表核心字段较少
- 足够支撑 `v1` 摘要，但不利于后续扩展

#### 改进任务

- 增加摘要版本信息
- 增加覆盖区间或来源范围信息
- 评估分层摘要是否必要
- 为后续回滚和重建预留字段

#### 验收标准

- summary 不再只是 `content + last_message_id`
- 数据结构能支撑更成熟的摘要生命周期

### 9. 将评估从 Retrieve 层扩展到 Answer 层

#### 当前问题

- 当前更强的是 retrieve 指标
- 对“最终回答质量提升”支撑不够

#### 改进任务

- 增加答案正确率评估
- 增加证据引用命中率评估
- 增加追问率或人工满意度统计
- 将 retrieval 增强和 answer 结果串联验证

#### 验收标准

- 能明确回答“检索提升是否真的转化成最终问答效果提升”

---

## P2：中期补强

### 10. 验证混合检索各通道真实贡献

#### 当前问题

- 检索框架是多通道的
- 但不同通道的真实收益不够透明
- 容易出现“框架是 hybrid，效果全靠 semantic”的情况

#### 改进任务

- 分析 `vector_global / keyword / metadata_title / memory_fact` 的命中贡献
- 区分哪些通道是有效增益，哪些只是结构存在
- 调整默认搜索模式和权重策略

#### 验收标准

- 每条通道的价值可以量化
- 默认混合模式有数据支撑，而不是先验假设

### 11. 升级 SubQuestion 策略

#### 当前问题

- 现在能拆，也能并行
- 但对子问题依赖关系的建模仍偏启发式
- 串并切换主要靠 dependency risk 规则

#### 改进任务

- 限制拆分数量
- 区分独立检索子问题与依赖型子问题
- 优化串并切换规则
- 对 merge 冲突结果增加处理策略

#### 验收标准

- 可以清楚解释“为什么这个问题要并行，那个问题要串行”
- 并行不是默认炫技，而是有明确收益和边界

### 12. 收敛“用户端无感知”表述到真实能力

#### 当前问题

- 模型熔断、fallback、首包探测都已经有
- 但“完全无感知”这个说法偏满

#### 改进任务

- 区分链路连续性、首包体验、质量稳定性、风格一致性四类指标
- 补监控数据支撑“无感知”的具体层次

#### 验收标准

- 不再泛泛地说“用户端无感知”
- 能准确说出当前到底做到了哪一层无感知

### 13. 把 Ingestion 从后端编排能力补成完整产品能力

#### 当前问题

- 后端 workflow 执行和节点编排能力已具备
- 但“可视化连线配置”容易被理解成完整图形化产品能力

#### 改进任务

- 明确前端可视化编排范围
- 明确节点输入输出契约
- 明确任务运行可视化、节点日志和排障链路

#### 验收标准

- 面试时可以明确说清楚：当前做到的是后端编排引擎，还是完整可视化流水线平台

### 14. 为 Agent / Tool 建立任务级评估

#### 当前问题

- tool runtime 和 Plan-Act-Observe 已实现
- 但 agent 成效更偏功能性证明，缺少稳定任务评估

#### 改进任务

- 建立诊断类任务成功率评估
- 建立外部证据补充任务成功率评估
- 统计 agent 回合数、降级率、人工介入率
- 区分规则路径与 LLM 路径的适用边界

#### 验收标准

- 能量化 agent 的收益和边界
- 回答“不是只会跑 demo，而是能在实际任务上稳定工作”

---

## 跨模块共性任务

### 15. 让“能力存在”升级成“能力可证明”

#### 当前问题

- 多数模块都有功能闭环
- 但证明收益的证据分布不均衡

#### 改进任务

- 每项核心能力都补一组最小可行指标
- 区分：
  - 机制存在
  - 效果提升
  - 线上稳定

#### 验收标准

- 面试时每个亮点都能回答：
  - 做了什么
  - 怎么做的
  - 怎么证明有效
  - 边界在哪

### 16. 让“工程可用”升级成“产品收口”

#### 当前问题

- 当前很多能力处于迁移或补强过程
- 容易出现简历表述过满、源码实现偏 `v1` 的落差

#### 改进任务

- 为每个亮点定义当前成熟度
- 区分 `v1 可用 / v2 治理化 / v3 平台化`
- 避免对外表述超出代码现实

#### 验收标准

- 对内和对外描述一致
- 功能说法与实现边界匹配

---

## 优先级建议

最值得优先推进的五项任务：

1. 收口双 Agent Runtime
2. 为 Rewrite 增加强约束校验
3. 完善术语词典治理与统计
4. 把会话窗口和摘要改成 token-aware + async
5. 补 retrieve / answer / agent 三层评估闭环

---

## 一句话总结

当前项目最需要的，不是再堆更多功能点，而是把已有能力从：

- 能跑

升级成：

- 可控
- 可证
- 可统一
- 可持续演进

这才是后续把项目从“亮点很多但容易被追问穿”提升成“亮点扎实且站得住”的关键。
