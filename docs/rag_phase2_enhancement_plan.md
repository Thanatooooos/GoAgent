# RAG 二期增强方案

更新时间：2026-04-30

## 1. 背景与目标

当前项目中的 RAG 一期，目标是打通最小可用闭环，即：

- 会话管理
- 记忆装配
- 检索召回
- Prompt 组装
- 模型生成
- 消息落库
- 基础 trace

在一期完成后，系统将具备“可以完成一次基于知识库的问答”的能力。但这仍然主要解决的是“能答出来”，还没有真正解决复杂知识问答场景中的几个核心问题：

- 用户问题复杂时，单轮检索很难一次召回足够证据
- 问题表述模糊时，首轮 query 质量不稳定
- 多文档、多知识点、多步骤问题难以组织成高质量回答
- 检索结果即使命中，也可能因为上下文组织粗糙而没有被模型充分利用
- 用户很难理解“答案是怎么来的”，工程侧也难以持续调优

因此，RAG 二期的目标不是简单继续补接口，而是让系统从“基础 RAG”升级为“增强型 RAG”，进一步演进到具备一定自主搜索能力的 `Agentic RAG`。

一句话概括二期目标：

把系统从“单轮检索问答器”升级为“可多步搜索、可证据归因、可持续调优的知识助手”。

## 2. 二期总体定位

结合当前项目进展，建议将 RAG 能力演进分成三个层次：

### 2.1 基础 RAG

解决“答得出来”：

- 单轮 query rewrite
- 向量检索
- 可选 rerank
- Prompt 组装
- LLM 回答
- 消息与摘要持久化

这一层对应当前一期建设目标。

### 2.2 增强 RAG

解决“答得更准、更稳、更可控”：

- 混合召回
- 更高质量的上下文工程
- 检索结果去重、聚合、压缩
- 引用证据返回
- 检索与回答可观测
- 用户反馈闭环

### 2.3 Agentic RAG

解决“复杂问题也能自己找路”：

- 自动判断是否需要多步搜索
- 问题拆解与子任务规划
- 多轮检索与补充检索
- 中间证据汇总与冲突检查
- 结构化生成最终答案

这一层是二期最有价值的方向，但必须建立在增强 RAG 的基础上。

## 3. 核心设计原则

为了避免二期一开始就走向过重、过散、不可控的方向，建议遵循以下原则：

### 3.1 先做“轻 Agent”，不做“万能 Agent”

二期重点应放在 `Agent Search`，即围绕“搜索和证据组织”做增强，而不是一开始就放开大量外部工具调用。

建议初期只允许 Agent 使用以下能力：

- query rewrite
- retrieval
- rerank
- answer synthesis
- trace logging

暂不默认开放：

- 外部 HTTP 工具
- 任意脚本执行
- 数据写操作
- 业务系统控制类动作

### 3.2 先保证检索底座稳定，再叠加搜索策略

如果召回质量不稳定，多轮搜索只会放大错误。因此，二期不应把 Agent Search 作为唯一主线，而要先同步加强：

- chunk 质量
- metadata 结构
- hybrid retrieval
- context engineering

### 3.3 把“可观测”作为正式能力建设

二期很多能力都不是上线即定型，而是要持续调优。因此，必须把以下能力当成正式模块建设：

- trace
- recall logging
- rerank logging
- prompt logging
- feedback logging
- evaluation sample

### 3.4 将用户能力做成模式分层，而不是堆开关

相比暴露大量参数，更建议对用户提供清晰的能力模式：

- `Quick Ask`
- `Agent Search`
- `Research`

这样既方便产品表达，也有利于后端链路按模式演进。

## 4. 二期能力方向

## 4.1 方向一：Hybrid Retrieval 升级

### 目标

将当前以向量检索为主的模式升级为多路召回融合，提高召回稳定性与覆盖率。

### 建议能力

- 向量召回
- 关键词召回
- 标题召回
- metadata 过滤召回
- 多路结果融合
- rerank 统一排序

### 适用价值

- 对报错信息、接口名、版本号等关键词敏感内容更稳
- 对 FAQ、操作手册、说明文档的命中率更高
- 为后续 Agent Search 提供更稳定的底座

### 对现有项目的贴合点

当前已有：

- embedding 抽象
- vector store 检索能力
- rerank 插槽

二期可在此基础上补：

- 检索路由层
- 多路召回融合器
- 查询策略配置

### 推荐优先级

`P0`

## 4.2 方向二：Context Engineering 升级

### 目标

让“召回到了的内容”能以更适合模型消费的方式进入 Prompt，减少碎片化上下文对回答质量的损害。

### 建议能力

- 相邻 chunk 合并
- 重复 chunk 去重
- 按文档与主题聚合
- 核心证据与补充证据分层
- 长上下文压缩摘要
- 引用片段标准化输出

### 适用价值

- 降低“搜到了但没答好”的概率
- 提升长文档、多 chunk 场景下的回答连贯性
- 为 Research 模式提供更稳的证据组织能力

### 对现有项目的贴合点

当前已知风险中明确指出：

- chunk 语义质量仍较粗
- metadata 不够丰富

因此，这一方向与 chunk 优化是强耦合关系，应并行推进。

### 推荐优先级

`P0`

## 4.3 方向三：Agent Search

### 目标

在基础检索问答之上，引入轻量多步搜索机制，让系统能对复杂问题进行分解、补充检索与证据汇总。

### 核心理解

这里的 Agent Search 不等于通用 Agent，而是“围绕知识搜索过程做自主化增强”。

其最小闭环应包括：

1. 判断当前问题是否需要多步搜索
2. 生成一个初始搜索计划
3. 将复杂问题拆成若干子问题
4. 对每个子问题分别检索
5. 汇总中间证据并判断是否存在缺口
6. 必要时继续补充检索
7. 生成带引用的最终答案

### 典型适用场景

- 需要跨多个文档综合回答
- 需要比较多个方案或多个版本差异
- 需要回答“原因 + 影响 + 建议”类问题
- 需要按步骤排障、排查、诊断

### 为什么适合当前项目做二期主线

因为你们已经具备：

- conversation
- memory
- rewrite
- retrieve
- prompt
- rag trace

也就是说，Agent Search 缺的主要不是底层能力，而是编排层。

### 建议控制边界

建议二期先做 `Light Agent Search`：

- 只允许使用内部 retrieval 能力
- 只允许做只读型搜索
- 不引入外部执行工具
- 不在第一版做开放式 ReAct

### 推荐优先级

`P1`

这是二期最重要的产品升级方向。

## 4.4 方向四：Research Mode

### 目标

把 Agent Search 进一步产品化成“深度研究模式”，输出结构化、可追溯的分析结果。

### 建议能力

- 自动研究计划
- 多轮多角度检索
- 证据分组
- 冲突观点标记
- 结论 + 依据 + 不确定性说明
- 输出结构化报告

### 与 Agent Search 的关系

可以理解为：

- `Agent Search` 更偏回答复杂问题
- `Research Mode` 更偏生成研究型结果

前者像“多步问答”，后者像“面向分析任务的小型研究助手”。

### 产品价值

适合在同一个系统里区分：

- 快速问答
- 深度分析

这样既保留响应速度，也保留高价值复杂任务入口。

### 推荐优先级

`P2`

## 4.5 方向五：Memory-Aware RAG

### 目标

将现有会话记忆能力与知识检索深度结合，让系统不仅理解“资料里有什么”，也理解“这个用户当前到底在追问什么”。

### 建议能力

- 检索前结合会话摘要做 query rewrite
- 区分短期上下文与长期会话摘要
- 对多轮问题形成意图延续
- 对用户反馈做轻量偏好适配

### 产品价值

- 减少用户多轮追问时的重复表达成本
- 提升连续会话中的命中率与回答一致性
- 使系统更像“协作助手”，而非“单轮搜索器”

### 对现有项目的贴合点

当前 `core/memory`、`conversation_summary` 已经落地，这条线具备较好的接入条件。

### 推荐优先级

`P1`

## 4.6 方向六：Tool-Augmented RAG

### 目标

在知识检索之外，逐步开放受控工具调用能力，使系统从“回答问题”升级到“帮助完成任务”。

### 可能工具类型

- 数据库只读查询
- 配置查询
- 外部文档抓取
- API 状态查询
- 内部业务系统信息读取

### 风险

- 权限边界复杂
- 结果可信度控制更难
- 失败重试和超时恢复复杂
- 会显著提高编排复杂度

### 建议判断

这条线有价值，但不建议作为二期最前排主线，应在 Agent Search 稳定后逐步引入。

### 推荐优先级

`P2`

## 5. 推荐产品能力分层

为避免后续能力不断叠加后用户无法理解，建议二期开始就把产品能力设计成三档模式：

### 5.1 Quick Ask

定位：快速问答

特点：

- 单轮检索
- 快速返回
- 成本低
- 适合高频日常问题

### 5.2 Agent Search

定位：复杂问题搜索

特点：

- 支持问题拆解
- 支持多轮检索
- 支持证据汇总
- 适合复杂知识问答与排查分析

### 5.3 Research

定位：深度研究与报告输出

特点：

- 允许更长时延
- 输出结构化结论
- 强调证据链
- 强调不确定性说明

这种模式分层的好处是：

- 用户理解成本低
- 后端策略边界清晰
- 方便后续按模式控制时延、成本、检索深度与回答格式

## 6. 推荐实施路线图

## 6.1 P0：打稳增强 RAG 地基

### 目标

在基础 RAG 闭环完成后，优先建设支撑二期所有增强能力的底层能力。

### 建议事项

- 补齐 hybrid retrieval
- 优化 chunk metadata
- 增强 context engineering
- 标准化引用片段输出
- 完善 retrieval / rerank / prompt trace
- 建立评测样本与效果基线

### 预期收益

- 回答质量更稳
- 后续 Agent Search 不容易放大召回问题
- 工程侧具备调优抓手

## 6.2 P1：落地 Light Agent Search

### 目标

在受控边界内实现“多步检索问答”。

### 建议事项

- 增加 query complexity 判断
- 增加 search planner
- 增加 sub-query 执行器
- 增加 evidence aggregator
- 增加 iterative search 控制器
- 增加面向 Agent Search 的 trace 节点定义

### 预期收益

- 支持复杂问题
- 支持跨文档汇总
- 提升系统上限

## 6.3 P1：补上 Memory-Aware RAG

### 目标

让多轮会话中的检索与回答更连贯。

### 建议事项

- 检索前结合历史摘要做 query rewrite
- 引入对话上下文窗口策略
- 对连续追问做轻量意图承接

### 预期收益

- 提升连续对话体验
- 减少用户重复输入

## 6.4 P2：产品化 Research Mode

### 目标

形成高价值深度分析能力。

### 建议事项

- 输出结构化研究报告
- 统一结论/证据/不确定性模板
- 支持更长 trace 展示

## 6.5 P2：按需引入 Tool-Augmented RAG

### 目标

从“会搜索”逐步演进到“会辅助执行”。

### 建议事项

- 只开放白名单工具
- 只做只读工具优先
- 完善权限与审计日志

## 7. 建议的后端模块演进

结合当前代码结构，建议二期重点新增或扩展以下模块：

### 7.1 core 层

- `rag/core/retrieve`
  - 扩展为 hybrid retrieval 编排
- `rag/core/rewrite`
  - 增加多轮 rewrite 与 search-oriented rewrite
- `rag/core/context`
  - 新增上下文聚合、压缩、去重能力
- `rag/core/search`
  - 新增 Agent Search 编排抽象
- `rag/core/evidence`
  - 新增证据归并与引用组织能力

### 7.2 service 层

- `RagChatService`
  - 继续承担 Quick Ask 主链路
- `AgentSearchService`
  - 承担多步搜索执行
- `ResearchService`
  - 承担深度研究模式

### 7.3 trace 层

建议在现有 `rag_trace_run / rag_trace_node` 基础上补充节点类型，例如：

- query_rewrite
- recall_vector
- recall_keyword
- rerank
- context_merge
- sub_query_plan
- sub_query_execute
- evidence_aggregate
- final_answer

## 8. 主要风险与控制建议

### 风险一：Agent Search 复杂度过早失控

控制建议：

- 第一版只做内部检索 Agent
- 不引入开放式工具执行
- 限制最大搜索步数与最大子问题数

### 风险二：召回质量不稳导致多步搜索放大错误

控制建议：

- 先建设 hybrid retrieval
- 建立评测集
- 每条策略调整都做基线对比

### 风险三：trace 不够细，后续无法调优

控制建议：

- 把 trace 当正式能力建设
- 从二期开始统一节点模型

### 风险四：产品入口太多，用户理解成本高

控制建议：

- 收敛为 `Quick Ask / Agent Search / Research` 三档模式
- 不对用户暴露过多内部参数

## 9. 最终建议

如果只选一条最符合当前项目发展方向的二期主线，建议选择：

`增强 RAG 地基 + Light Agent Search`

原因是：

- 它最贴合你们当前已具备的 conversation、memory、retrieve、trace 基础
- 它比直接做通用 Agent 更可控
- 它比单纯继续优化 chunk 更容易形成用户可感知的产品升级
- 它能够自然衔接后续的 Research Mode 和 Tool-Augmented RAG

建议最终路线为：

1. 先完成基础 RAG 闭环
2. 再补强 hybrid retrieval 与 context engineering
3. 随后落地 Light Agent Search
4. 最后把 Agent Search 产品化为 Research 模式，并逐步接入受控工具

这一路线能够兼顾：

- 工程可落地性
- 产品可感知价值
- 后续可扩展空间
