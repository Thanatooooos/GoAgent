# 2026-05-24 工作记录

## Memory P0 加固与 P1 收口

### 今日目标

今天这轮工作没有开启新的 memory phase，而是集中修当前 `Phase 4+` 里的明显风险点，目标有两类：

1. 先把 `P0` 的治理正确性和降级闭环补齐
2. 再把 `P1` 的 recall 质量问题收掉一部分

重点覆盖：

- 单值记忆并发写坏
- 冲突检测依赖小窗口
- cache fallback 同请求重复计算
- 规则型记忆排序不符合治理语义
- `display_value` 误合并
- 事实型 recall 的 lexical prefilter 太脆

### 完成内容

#### 1. 单值记忆补上了数据库级唯一性保护

- 新增 migration：
  - `internal/adapter/repository/postgres/migrations/20260524110000_add_memory_item_single_active_unique_index.sql`
- 对单值 canonical key 的 `active` 记录增加唯一索引
- 索引按 `(user_id, scope_type, COALESCE(scope_id, ''), canonical_key)` 约束
- migration 里先做 duplicate-active 预检查，避免脏数据被静默带过去

这一步的结果是：单值 key 不再只靠应用层约定维持 "only one active"。

#### 2. 显式保存链路现在会对并发冲突做收敛

- `SaveExplicitMemory(...)` 现在识别单值唯一冲突
- 遇到并发写入时会：
  - 重读当前 active
  - 同值则返回已存在记录
  - 不同值但别人先写成功，则返回并发赢家
  - 如果发现历史上已经有多个 active，直接报治理异常

这一步的结果是：并发双写不再轻易制造更多坏数据。

#### 3. 冲突检测不再依赖 `Limit: 8`

- repo 增加了精确查询当前 `(user, scope, canonical_key)` active 记录的能力
- 也增加了单值 active 冲突扫描能力，后续可复用于巡检/修复
- `conflict_detector` 不再靠“最近几条 active”做治理判断

这一步的结果是：治理逻辑不再随着历史数据变脏而失真。

#### 4. request-scope cache fallback 闭环补齐

- 在 long-term memory recall 里：
  - scope version unavailable
  - Redis disabled
  - Redis fallback
  这些路径现在也会写回 request cache
- 保留原来的 `fail-open`
- 但降级时不会在同一 request 内反复重算 recall / ranking

这一步的结果是：可用性和性能语义终于对齐了。

#### 5. 规则型记忆排序改成更符合治理语义

- 规则型记忆不再只是按 `update_time desc`
- 当前顺序稳定为：
  1. `scope priority`
  2. `importance`
  3. `last_confirmed_at`
  4. `update_time`
- request cache / Redis cache 命中路径也统一复用同样排序

这一步的结果是：真正重要的规则记忆更稳定地进入 `MemoryContext`。

#### 6. 等值判断收紧，降低误合并风险

- `memoryItemsEquivalent(...)` 现在优先比较：
  - 结构化值
  - `content`
  - 最后才回退到 `display_value`

这一步的结果是：不会再因为两个事实都显示成同一个短标签，就被错误地 merge 成一条。

#### 7. 事实型 recall 升级为 token 化 lexical prefilter

- `MemoryItemListFilter` 新增：
  - `SearchTokens`
- fact recall 现在会同时传：
  - `SearchText`
  - `SearchTokens`
- token 构造支持：
  - 英文词
  - 连续 CJK bigram
  - mixed-language query
- repo SQL 从单一整句 `%query%` 升级为：
  - 原始 query 兜底
  - token-based `OR` 预过滤

这一步的结果是：词序变化和混合语言问法的候选误杀明显降低。

#### 8. token 去噪与评分逻辑也对齐了

- 加了轻量 stopword/noise 过滤
- 英文过滤示例：
  - `how`
  - `should`
  - `the`
  - `please`
- 中文过滤示例：
  - `请问`
  - `这个`
  - `可以`
  - `怎么`
  - `了吗`
- `scoreMemoryText(...)` 也改成复用同一套 token 逻辑

这一步的结果是：DB 预过滤和应用层排序不会再一边去噪、一边吃噪音词。
