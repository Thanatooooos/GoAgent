### 完成内容

#### Phase 1: 长消息处理（100%）

按照 `docs/memory_architecture_design.md` 的 Section 5 设计，落地了完整的长消息三档处理链路。

**数据模型变更**：
- `domain.ConversationMessage` 新增 `RawContent / ContentSummary / IsSummarized` 三个字段
- `models.ConversationMessageModel`、mapper、前端 VO 同步对齐
- SQL migration: `20260519120000_add_message_summary_fields.sql`（ALTER TABLE t_message ADD COLUMN IF NOT EXISTS）

**核心实现**：
- 新增 `LongMessageContentProcessor`（`internal/app/rag/service/long_message_content_processor.go`，517 行）
  - `RoughTokenEstimator`：CJK/ASCII/Other 三分类粗略 token 估算
  - 三档阈值策略：
    - `< 3000 tokens`：直接进入上下文，不做处理
    - `3000 ~ 12000 tokens`：LLM 生成短摘要 → 规则 fallback（`buildMediumMessageSummary`）
    - `> 12000 tokens`：先切 chunk → 每 chunk LLM 摘要 → 合并总摘要（分层摘要）
  - `splitTextByTokenBudget`：按 token 预算切分，支持 overlap 保证跨 chunk 语义连续性
  - `detectLongMessageKind`：自动识别日志/错误/代码/文本四种类型
  - 摘要 prompt 按类型不同（medium/chunk/merge 三套 prompt）

**服务层改造**：
- `ConversationMessageService.AddMessage` 重构为 processor 模式：
  - 新增 `ConversationMessageContentProcessor` 接口（策略注入点）
  - 新增 `ProcessedConversationMessageContent` 返回结构（含 SessionChunks）
  - processor 返回的 `Content` 作为最终入库的 `content`，`RawContent` 保存原文
  - `AddConversationMessageInput` 也新增三字段，支持上游直接送入已处理结果

**配置**：
- `config.go` 新增 `RagLongMessageConfig`（8 个配置项，均有默认值兜底）
- `application.yaml` 新增 `rag.memory.long-message` 配置块

**前端 API**：
- `handlers.go` 的 `messageVO` 和 `toMessageVO` 同步补充新字段（`rawContent / contentSummary / isSummarized`）
