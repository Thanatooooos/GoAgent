# 2026-05-23 工作记录

## Memory V1 读路径分治与召回收口

### 今日目标

把 `memory V1` 从“Phase 2.1 治理闭环已完成，但读取路径还没有完全按设计分治”推进到“读路径更符合设计、召回更抗规模、数据更不易腐化”的状态。

本轮不做完整 `Phase 3` 检索投影，只做：

- 规则型 / 事实型记忆读路径分治
- DB 层 recall 预过滤
- `last_used_at` 运行时接入
- embedding 告警与 JSON 等值判断补强

### 完成内容

#### 1. 长期记忆 recall 已拆成规则型与事实型两条路径

- `internal/app/rag/service/longtermmemory/recall_service.go`
  - `memory_type=preference` 改为规则型直读
  - `memory_type=knowledge` 保留事实型 recall / rerank / topK
  - `memory_type=feedback` 当前默认退出 chat recall
- `RecallMemories(...)` 现在会先装载规则型记忆，再装载事实型候选，最后合成统一 `MemoryContext`

效果：

- 用户偏好不再和项目事实竞争统一 TopK
- 长期记忆读取路径首次真正符合 memory 设计文档里的分治要求

#### 2. `MemoryContext` 已改为分段输出

- `internal/app/rag/service/longtermmemory/context_renderer.go`
  - 新增 `Rule Memories`
  - 新增 `Fact Memories`
  - 每段内继续按 `KB-Scoped Memories / Global Memories` 渲染

效果：

- prompt 中的长期记忆语义边界更清晰
- 规则约束与事实细节不再混成一个列表

#### 3. 事实型 recall 新增 DB 层预过滤

- `internal/app/rag/port/repository.go`
  - `MemoryItemListFilter` 新增 `SearchText`
- `internal/adapter/repository/postgres/rag/memory_item_repo.go`
  - `List(...)` 新增 `ILIKE` 预过滤：
    - `summary`
    - `content`
    - `display_value`
    - `canonical_key`

效果：

- 事实型 recall 不再只靠应用层排序缩圈
- 对 active memories 的扫描压力开始下降

#### 4. `last_used_at` 已接入运行时

- `MemoryItemRepository` 新增：
  - `TouchLastUsed(ctx, userID, ids, at)`
- recall 选中 memory 后，会 best-effort 批量更新 `last_used_at`
- 更新失败只记 warning，不阻断 chat

效果：

- `last_used_at` 不再只是 schema 字段
- 为后续 cleanup / lifecycle / maintenance 做好了信号铺垫

#### 5. 补掉两个数据腐化点

- `internal/app/rag/service/longtermmemory/conflict_detector.go`
  - `value_type=json` 的等值判断改为结构化比较
  - JSON 字段顺序差异不再误触发 `superseded`
- `internal/app/rag/service/longtermmemory/service.go`
  - embedding 生成失败改为 `Warn`
  - embedding upsert 失败改为 `Warn`
  - 仍保持 fails-open

效果：

- 单值键不会因 JSON key 顺序差异制造无意义版本链
- embedding 丢失不再完全不可见

#### 6. trace 与测试已同步收口

- `internal/app/rag/service/rag_chat_prepare.go`
  - long-term memory stage extraData 新增：
    - `ruleCount`
    - `factCandidateCount`
    - `factSelectedCount`
    - `ruleMemoryIds`
    - `factMemoryIds`
- 补充并更新了：
  - `longtermmemory` 单测
  - `rag_chat_service` 集成测试
  - Postgres repo 测试
  - HTTP handler 测试辅助 stub

## Additional Update: Memory V1 Phase 3 最小闭环 + Phase 4 Cache Skeleton

### 本轮增量目标

在前半段读路径收口的基础上，继续把 `memory V1` 往前推两步：

1. 完成 `Phase 3` 的最小 retrieval projection
2. 给 `Phase 4` 的 cache 建设落下第一版骨架

这轮的策略不是把 memory 做“大而全”，而是先确保：

- `knowledge` 型长期记忆能以独立 channel 进入主 retrieve
- 后续 cache 不会挂在 chat 层，而是沉到 `longtermmemory` recall 层

### 完成内容

#### 1. `memory_fact` channel 已接入主 retrieve

- `internal/app/rag/core/retrieve`
  - 新增 `memory_fact` channel
  - 参与现有 `fusion / dedup / rerank`
- `internal/app/rag/service/longtermmemory/retrieve_projection.go`
  - `knowledge` 型长期记忆已可投影成 `RetrievedChunk`
- `RagChatService.prepareChat(...)`
  - retrieve 请求已把 `userID` 传下去

结果：
- 文档 miss 时，长期事实记忆可以救回 retrieve
- 文档与 memory 同命中时，文档仍优先
- `preference` 不进 retrieve
- `feedback` 不进 retrieve

#### 2. Phase 3 样例与评估样本集已补齐第一版

- 已补 retrieval-style 回归样例，覆盖：
  - `doc miss, memory hit`

  - `main_bus removed`

- 已新增样本集：
  - `testdata/memory_fact_phase3_samples.json`

#### 3. `memory_fact` 第一版融合权重已落定

当前默认权重：

- `vector_global = 1.0`
- `memory_fact = 0.9`
- `keyword = 0.85`
- `metadata_title = 0.8`

这保证了：
- 长期事实记忆已进入主检索
- 但第一版不会轻易压过主文档语义检索

#### 4. Phase 4 cache skeleton 已开始落地

- 已新增 Redis 导向的 `RecallCache` 抽象
- 已接入三类必做缓存骨架：
  - `rule memories`

  - 成功后会 bump 对应 version

  - 已支持按配置装配 Redis recall cache

当前有意暂未完成：
- request-scope `L1` cache
- session recall cache
- cache metrics

## Additional Update: Memory V1 Phase 4 Cache Closure

### 本轮完成目标

把今天前半段留下的 `Phase 4 cache skeleton` 真正补成完整闭环，并把 `session recall` 一起拉进同一套缓存与观测框架。

### 完成内容

#### 1. request-scope `L1` 已接入 `prepareChat(...)`

- 新增共享请求级缓存
- 在单次 `prepareChat(...)` 生命周期内复用：
  - `rule memories`
  - `fact ranking result`
  - `query embedding`
  - `session recall` 请求内结果

结果：

- `RecallMemories(...)` 与 `SearchFacts(...)` 不再在同一轮 chat 中重复做 fact ranking
- `session recall` 也能避免同请求内重复 embedding / recall

#### 2. conversation-scope `L1` 已用于 `session recall`

- 新增进程内 `TTL + LRU` 会话级缓存
- 当前只缓存最终 `SessionRecallResult`
- 空结果允许短 TTL 缓存

同时新增 recall fingerprint，用于在会话内容变化时自动失效：

- recallable chunk 是否存在
- recallable chunk 数量
- 最新更新时间
- 最新 chunk / message 标识

#### 3. 长期记忆 Redis `L2` 闭环已补齐

- Redis 继续承载：
  - `rule memories`
  - `fact ranking result`
  - `query embedding`
- 保留 `global / kb` scope version
- `SaveExplicitMemory(...)` / `ExpireMemory(...)` 成功后继续 bump version
- 新增 `redis-key-prefix` 配置

#### 4. trace / metrics / fail-open 已补齐

- 新增 `rag.memory.cache.*` 配置项
- 新增 `GET /rag/memory/metrics`
- `long_term_memory` trace 已补 cache layer / versions / recompute reason
- `session_recall` trace 已补 cache layer / fingerprint / recompute reason
- 所有缓存异常统一 `fail-open`
  - Redis get/set 异常
  - Redis 反序列化异常
  - fingerprint 查询异常
  - local cache evict / overflow
  都只允许降级，不阻断 chat / retrieve 主链路
