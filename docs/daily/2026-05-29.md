# 2026-05-29 工作记录

## Eino-Native Agent PoC：补齐 `fetch / webfetch`

### 今日目标

今天这轮工作的目标很聚焦：

1. 延续已经新建的 `internal/app/agent` 独立实验包，继续按 `search / websearch` 的双层拆分方式补齐 `fetch / webfetch`
2. 让新的 Eino-native agent 不再只是 `web_search` 单工具 PoC，而是能实际跑通最小的 `search -> fetch -> observe` 闭环
3. 保持这个新包完全独立于旧 `internal/app/rag/tool` 契约，只复用底层搜索 provider 和网页抓取能力的工程经验，不做旧框架兼容层

今天没有把这个新 agent 接回旧 `RagChatService`，也没有开始替换自研 `AgentLoop`。重点仍然是先验证“用 Eino 原生组织 agent / tool / state”这条路是否顺手、是否清晰。

### 完成内容

#### 1. `internal/app/agent` 继续按双层职责补齐了 `fetch / webfetch`

新增：

- `internal/app/agent/fetch`
  - `service.go`
  - `extract.go`
  - `types.go`
- `internal/app/agent/webfetch`
  - `tool.go`

职责边界保持和 `search / websearch` 一致：

- `fetch`
  - 负责抓取业务内核
  - URL 规范化
  - 并发抓取
  - HTML 正文提取
  - 文本截断
  - 页面级结果汇总
- `webfetch`
  - 负责把 `fetch.Service` 暴露成 Eino 原生 `web_fetch` tool
  - 只拥有 tool schema、输入映射和输出映射

这样新 agent 包里现在已经有两组清晰对称的能力边界：

- `search / websearch`
- `fetch / webfetch`

#### 2. Eino-native workflow 从“单搜索”升级成最小外部证据链

`internal/app/agent/workflow` 这轮的主要变化是：

- 原来的 `act_agent.go` 拆成：
  - `search_agent.go`
  - `fetch_agent.go`
- `builder.go` 现在组装的是：
  - `plan`
  - `search`
  - `fetch`
  - `observe`

也就是说，新 workflow 已经不再只是：

- `plan -> web_search -> observe`

而是变成：

- `plan -> web_search -> web_fetch -> observe`

同时补了新的 session state：

- `agent.search.output`
- `agent.fetch.output`

`observe_agent.go` 也同步升级成会综合判断：

- search 是否 degraded
- fetch 是否 degraded
- fetch 是否拿到了可读正文

最终再产出统一的 `FinalState`。

#### 3. `agent.Service` 的对外响应已经能返回抓取内容

`internal/app/agent/service.go` 和 `request_response.go` 这轮同步更新了对外接口：

- `Response.Results`
  - 保留搜索结果列表
- `Response.Pages`
  - 新增抓取后的页面列表
- `Response.CombinedText`
  - 新增拼接后的可读正文

这样新的 `internal/app/agent` 已经具备一个更完整的最小应用层入口：

- 输入 `Question`
- 内部执行 `search -> fetch`
- 输出搜索结果、抓取页面和汇总文本

#### 4. 抓取内核独立成包后，后续扩展空间更清晰了

这一步的意义不只是多了一个 tool，而是把后续扩展的骨架定住了：

- 如果下一步要补 `URL 选择策略`
  - 优先 `allow`
  - 限制抓取数量
  - 过滤高风险域名
  就放在 `workflow` 和 `fetch` 之间的编排层处理
- 如果下一步要补更复杂的正文抽取或质量评估
  就继续收敛在 `fetch` 包内
- 如果未来要让 Eino loop 直接继续长成更完整的 agent runtime
  当前 `search / websearch / fetch / webfetch / workflow` 这套边界可以直接延展，不需要回头重拆

### 今日结论

今天之后，`Agent / Tool` 这条线多了一个新的明确信号：

1. Eino-native agent 已经不再只是“讨论中的方向”，而是落成了一个独立、可测试、可继续扩展的实验骨架
2. 这条实验线当前是“新包并行推进”，而不是“回头兼容旧框架”或“直接替换主链”
3. 当前最合理的下一步，不是立刻重写全部 agent loop，而是继续在这个新包里把：
   - 抓取目标选择
   - 页面质量判断
   - 多工具编排语义
   做得更像真实 agent

接下来更合理的顺序是：

1. 给 `web_search -> web_fetch` 之间补更明确的 URL 选择策略
2. 评估 `fetch` 结果质量和终止条件，而不是只按“抓到了就结束”
3. 再决定是否继续往这个新包里补更多原生 tool
4. 等 PoC 结论更稳定后，再讨论是否把它作为未来替换自研 `AgentLoop` 的候选路径
