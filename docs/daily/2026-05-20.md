## Memory V1 会话检索层闭环

### 今日目标

把 `memory V1` 的 `Phase 1.5` 从“存储侧完成、召回侧待做”推进到真正可用的会话检索闭环：让当前 `conversation_id` 内的长 `user` 消息原文能在后续轮次被自动轻量召回，并单独注入 prompt。

### 完成内容

#### 1. 修正了长消息写入覆盖范围

- `LongMessageContentProcessor` 现在不仅为超长消息生成 `SessionChunks`，也会为 `3000~12000 token` 的中等长消息生成 `SessionChunks`
- 这样实现重新与设计结论对齐：
  - 中等长消息：摘要进上下文，原文 chunk 进入会话检索层
  - 超长消息：chunk 摘要后合并总摘要，原文 chunk 进入会话检索层

#### 2. 落地了 session recall 查询与服务

- `SessionChunkRepository` 新增：
  - `ExistsRecallable(...)`
  - `SearchRecallableByVector(...)`
- 新增 `SessionRecallService`
  - 作用域固定为当前 `conversation_id`
  - 仅召回 `user` 且 `is_summarized=true` 的历史长消息
  - 排除当前轮刚写入的消息
  - 先判断是否存在 recallable chunk，再决定是否执行 query embedding

#### 3. 接入了 `prepareChat` 主链路

- `RagChatService.prepareChat(...)` 现在按顺序执行：
  - `runMemoryStage`
  - `runUserMessageStage`
  - `runRuntimeStage`
  - `runRewriteStage`
  - `runSessionRecallStage`
  - `runRetrieveStage`
- `session_recall` 走独立旁路，不影响现有 KB retrieve 判定
- 召回失败按 fail-open 处理，不阻断 chat 主链路

#### 4. Prompt 与 trace 已形成闭环

- prompt 新增独立 `SessionContext`
  - 以 `## 会话上下文片段` system section 注入
  - 不混入 history，也不并入 `KnowledgeContext`
- excerpt 选择已按轻量 lexical overlap 落地
  - 小 chunk 直接回放完整原文
  - 大 chunk 二次切窗后再择优
  - 最终固定输出“摘要 + 原文 excerpt”
- `session_recall` 已有独立 trace node：
  - `node_id=session_recall`
  - `node_type=memory`
  - `node_name=session_chunk_recall`
- trace extraData 现可表达：
  - `used`
  - `candidateCount`
  - `excerptCount`
  - `topScore`
  - `excludedMessageId`
  - `selectedHits`
  - `skippedPerMessageLimit`
  - `truncatedBy`

#### 5. 补了两类近似真实场景样例

- 长日志追问：
  - 前一轮贴长日志
  - 下一轮追问 `retriever timeout`
  - 验证细节可从前一轮日志中补回
- 长配置追问：
  - 前一轮贴较长配置
  - 下一轮追问 `web-search provider`
  - 验证 `provider: tavily-mcp` 之类配置细节可从前一轮配置中补回

## Memory V1 Phase 2 基础闭环

### 今日目标

把 `memory V1` 从“Phase 1.5 已完成、Phase 2 只停留在设计”推进到“显式长期记忆基础闭环可用”：

- 能保存长期记忆
- 能按 `global / kb` 组织
- 能在后续聊天前被轻量 recall
- 能以独立 prompt section 参与回答

### 完成内容

#### 1. 落地了 `memory_item` 主模型

- 新增 `domain.MemoryItem`
- 新增 migration：
  - `20260520120000_create_memory_item_table.sql`
- 新增持久化模型与仓储：
  - `MemoryItemModel`
  - `MemoryItemRepository`
- 这一层现在承载长期记忆的业务真相，而不是继续借用普通 `knowledge document`

#### 2. 落地了 `MemoryService`

- 新增：
  - `SaveExplicitMemory(...)`
  - `ListMemories(...)`
  - `ExpireMemory(...)`
  - `RecallMemories(...)`
- 当前默认语义：
  - `scope_type` 默认 `global`
  - `memory_type` 默认 `knowledge`
  - 自动生成 `summary`
  - 显式保存后直接写成 `status=active`

#### 3. 接入了 chat prepare 链路

- `RagChatService` 新增独立 `long_term_memory` stage
- `prepareChat(...)` 当前关键顺序变为：
  - `runMemoryStage`
  - `runUserMessageStage`
  - `runRuntimeStage`
  - `runRewriteStage`
  - `runLongTermMemoryStage`
  - `runSessionRecallStage`
  - `runRetrieveStage`
- 这意味着长期记忆现在已经可以在后续对话中被消费，而不是只停留在数据库记录层

#### 4. Prompt 分层继续保持清晰

- `ragprompt.Context` 新增 `MemoryContext`
- 长期记忆以独立 section 注入 prompt
- 不混入：
  - `History`
  - `SessionContext`
  - `KnowledgeContext`

这样做的好处是：
- 短期记忆、会话检索、长期记忆三层语义继续分离
- 后续做 Phase 3 检索投影时不会和现有 prompt 结构缠在一起

#### 5. HTTP 与 runtime wiring 已补齐

- 新增接口：
  - `POST /rag/v3/remember`
  - `POST /rag/v3/memories`
  - `GET /rag/v3/memories`
  - `POST /rag/v3/memories/:memoryId/expire`
- `bootstrap/rag.Runtime` 已接入：
  - `MemoryItemRepository`
  - `MemoryService`
  - `RagChatService.SetExplicitMemoryService(...)`

### 当前边界

这轮仍然是 **Phase 2**，不是 **Phase 3**。

当前 recall 还是轻量实现：
- 先按 `kb` / `global` 作用域取数
- 再做轻量 lexical match 与优先级排序

还没有做：
- `memory_item -> 检索投影`
- 向量化长期记忆召回
- 异步自动识别
